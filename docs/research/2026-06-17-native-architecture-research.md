## Menu-bar app shell & lifecycle (SwiftUI MenuBarExtra, accessory app, SMAppService launch-at-login, App Sandbox decision + entitlements, Developer ID code-signing/notarization, deployment target)

**Summary.** Build Daybrief as a SwiftUI App with a single MenuBarExtra scene using .menuBarExtraStyle(.window) to host the rich brief panel, plus a separate Window/Settings scene for onboarding, connector OAuth, and preferences. Make it an accessory (menu-bar-only, no Dock icon) — prefer programmatic NSApplication.setActivationPolicy(.accessory) over the static LSUIElement Info.plist key so the app can temporarily become .regular to bring its onboarding/settings window forward, then drop back to accessory. Use SMAppService.mainApp for launch-at-login (register/unregister + status, with the Login Items approval flow surfaced in UI). The load-bearing decision is App Sandbox: because Google's "Desktop app" OAuth client type requires the loopback (127.0.0.1 + ephemeral local HTTP listener) redirect flow and does NOT support custom URI schemes, and because sandbox + a loopback listener is fragile under Developer ID, ship UNSANDBOXED with a hardened runtime for the v0 outside-MAS Developer ID build (sandbox is optional for Developer ID and only mandatory for the Mac App Store). Sign with Developer ID Application + hardened runtime, notarize with notarytool, staple with xcrun stapler, ship in a notarized+stapled DMG. Recommended minimum deployment target: macOS 14 Sonoma (broadest install base while keeping modern SMAppService/SwiftUI), unless the team wants to hard-require macOS 26 Tahoe Liquid Glass — then target 26.


**Decisions**
- **Single MenuBarExtra scene with .menuBarExtraStyle(.window) for the brief panel; a separate WindowGroup/Window scene (or Settings scene) for onboarding, OAuth, and preferences.** — The .window style renders a popover-like panel that supports arbitrary rich SwiftUI (scrollable brief, sections, buttons, images), whereas .menu silently drops images/custom button styles and is limited to text/buttons/dividers. The brief is data-rich, so .window is required. OAuth and onboarding need a real, resizable, focusable window (web auth sheet, account list), which a transient popover cannot host reliably — hence a second scene.
- **Accessory (menu-bar-only) app via NSApplication.shared.setActivationPolicy(.accessory) set at launch, with temporary promotion to .regular when showing the onboarding/settings window, then demotion back to .accessory on close.** — Daybrief should not occupy the Dock or app switcher in steady state. Pure static LSUIElement=YES (a.k.a. Application is agent) hides the Dock icon but makes it hard to reliably front a settings/onboarding window and give it keyboard focus from a menu-bar-only app — a well-documented SwiftUI pain point. Programmatic activation policy lets the app live as accessory but momentarily become a regular app to present modal-feeling onboarding/OAuth windows with proper focus and menu bar, then return to accessory. This avoids the 'window appears behind other apps / no focus' class of bugs.
- **Launch-at-login with SMAppService.mainApp.register()/unregister(); read .status to drive the toggle; surface the Login Items approval path when status == .requiresApproval.** — SMAppService (macOS 13+) is the current, non-deprecated API (replaces SMLoginItemSetEnabled and helper-bundle hacks). For a main-app login item you use SMAppService.mainApp — no embedded helper, no extra plist. After register(), status can be .requiresApproval (user must enable in System Settings > General > Login Items), so the UI must read status (don't store a local bool) and offer SMAppService.openSystemSettingsLoginItems() to guide the user. unregister() cleanly removes it. No entitlement is required for mainApp login-item registration.
- **Ship the v0 Developer-ID (outside-MAS) build UNSANDBOXED, with hardened runtime enabled.** — App Sandbox is mandatory only for Mac App Store distribution; it is optional for Developer ID. Daybrief's Google connector must use the Google 'Desktop app' OAuth client, which requires the loopback redirect flow (127.0.0.1 + ephemeral local HTTP listener) and does NOT support custom URI schemes. Running a loopback HTTP listener under sandbox requires com.apple.security.network.server and is a known source of fragility/violations under Developer ID. Skipping the sandbox removes that whole risk surface for an OSS local-first tool, while hardened runtime still satisfies notarization. If MAS distribution is ever wanted, revisit with sandbox + network.server + ASWebAuthenticationSession-only providers.
- **Prefer ASWebAuthenticationSession for any provider that supports a custom-scheme/HTTPS app redirect (likely Slack, and Gmail/Calendar IF using an iOS/genuine app client), but keep a loopback (127.0.0.1 ephemeral-port) HTTP listener for the Google Desktop-app client.** — ASWebAuthenticationSession is the privacy-preserving, system-managed flow with the best UX (no 'Open' prompt) and needs no listening socket. But Google explicitly dropped custom-scheme support for the Desktop OAuth client type and continues to support loopback only for desktop clients — so a loopback listener is unavoidable for the Google connectors as specified. Design the Connector protocol so each connector declares its redirect strategy (ASWebAuthenticationSession vs loopback) rather than assuming one.
- **Code-sign with Developer ID Application + --options runtime (hardened runtime), notarize via xcrun notarytool submit --wait, then xcrun stapler staple, and distribute a notarized+stapled DMG (signed with Developer ID).** — This is the standard, current path for distributing outside the Mac App Store. notarytool is the supported submission tool (the legacy altool path is gone). Stapling the ticket to both the .app and the enclosing DMG lets Gatekeeper validate offline on first launch. Hardened runtime is a hard precondition of notarization.
- **Minimum deployment target: macOS 14 Sonoma (fallback default); choose macOS 26 Tahoe only if Liquid Glass / Tahoe-specific MenuBarExtra polish is a hard product requirement.** — SMAppService.mainApp and MenuBarExtra(.window) both exist from macOS 13, and Swift 6.2 strict concurrency / Xcode 26 toolchains comfortably target macOS 14. Sonoma maximizes addressable install base for an OSS tool while losing nothing the shell needs. Targeting 26 buys the newest SwiftUI/Liquid Glass affordances but strands every user not yet on Tahoe — only worth it if the team commits to Tahoe-only.

**Stack**
- `SwiftUI MenuBarExtra` (macOS 13+ (use .window style)) — Native menu-bar scene; .window style hosts the rich brief panel. First-party, no third-party menu-bar lib needed for the core.
- `ServiceManagement / SMAppService` (macOS 13+) — Current launch-at-login API (SMAppService.mainApp). Replaces deprecated SMLoginItemSetEnabled.
- `AuthenticationServices / ASWebAuthenticationSession` (macOS 10.15+) — Privacy-preserving system OAuth flow for providers that allow a custom-scheme/app redirect (no listening socket).
- `Network framework loopback HTTP listener (NWListener) or Swift NIO` (macOS 14+ baseline) — Required for Google Desktop-app OAuth client which mandates the 127.0.0.1 loopback redirect; bind an ephemeral port on 127.0.0.1, capture the auth code, shut down immediately.
- `Security framework (Keychain Services)` (macOS baseline) — Stores the SQLCipher DB key and OAuth tokens. No keychain-access-groups entitlement needed for single-app keychain use; unsandboxed app uses the login keychain directly.
- `notarytool + stapler (Xcode 26 command-line tools)` (Xcode 26) — Notarization submission and ticket stapling for Developer ID DMG distribution.
- `Developer ID Application certificate + hardened runtime` (current) — Required to sign and notarize an app distributed outside the Mac App Store.

**Patterns / concrete notes**
- App entry: @main struct DaybriefApp: App { var body: some Scene { MenuBarExtra('Daybrief', systemImage: 'sun.max') { BriefPanelView() }.menuBarExtraStyle(.window); Window('Daybrief Settings', id: 'settings') { SettingsView() } } } — keep the brief in MenuBarExtra(.window) and onboarding/OAuth/prefs in a separate Window scene.
- Accessory + temporary promotion: in an NSApplicationDelegateAdaptor's applicationDidFinishLaunching call NSApp.setActivationPolicy(.accessory). When opening the settings/onboarding/OAuth window, call NSApp.setActivationPolicy(.regular) + NSApp.activate(ignoringOtherApps: true); on window close, return to .accessory. This fixes the 'settings window won't focus from a menu-bar app' problem.
- Launch-at-login toggle bound to SMAppService: read let svc = SMAppService.mainApp; switch svc.status { case .enabled / .requiresApproval / .notRegistered / .notFound }. On enable: try svc.register(); if it throws or status becomes .requiresApproval, call SMAppService.openSystemSettingsLoginItems() and show guidance. On disable: try svc.unregister(). Drive the SwiftUI Toggle off svc.status, never a persisted bool.
- Per-connector redirect strategy: define in the Connector protocol an enum RedirectStrategy { case asWebAuth(callbackScheme: String); case loopback } so Google uses .loopback (NWListener on 127.0.0.1:0, hand the captured ?code= back, then cancel the listener) and providers that allow it use ASWebAuthenticationSession with a registered URL scheme (CFBundleURLTypes).
- Loopback listener lifecycle: bind NWListener to 127.0.0.1 on port 0 (OS-assigned ephemeral port), read the assigned port to build the redirect_uri, start ASWebAuthenticationSession or open the default browser, capture exactly one GET with ?code=&state=, validate state, then immediately listener.cancel(). Keep it alive only for the duration of one auth.
- Notarization pipeline (CI/Makefile): codesign --deep --force --options runtime --timestamp --sign 'Developer ID Application: <Team>' Daybrief.app; build DMG; codesign the DMG with the same identity; xcrun notarytool submit Daybrief.dmg --keychain-profile <profile> --wait; xcrun stapler staple Daybrief.dmg (and staple the .app before packaging). Use a notarytool keychain profile (store the App Store Connect API key / app-specific password once) so CI is non-interactive.
- Entitlements file even when unsandboxed: still ship an .entitlements with com.apple.security.cs.allow-jit / disable-library-validation OFF by default (don't loosen hardened runtime unless a dependency forces it). For unsandboxed Developer ID you generally need NO sandbox network entitlements; keychain works against the login keychain directly.
- If sandbox is later required (MAS path): add com.apple.security.app-sandbox=true, com.apple.security.network.client=true (outbound LLM/API calls), com.apple.security.network.server=true ONLY if keeping the loopback listener, and prefer dropping the Google Desktop-client/loopback flow in favor of an OAuth flow that works with ASWebAuthenticationSession to avoid network.server.

**Gotchas**
- MenuBarExtra(.window) has no first-party API to programmatically dismiss the panel after an action (open feedback FB11984872). A control inside the panel cannot close it via a binding on stock SwiftUI. Plan for this: either accept user-driven dismissal, or pull in a helper (e.g. orchetect/MenuBarExtraAccess) to access the underlying NSStatusItem/NSWindow and toggle presentation.
- MenuBarExtra(.menu) blocks the run loop while open, so isPresented-style bindings don't update until the user dismisses the menu — another reason to use .window for anything interactive.
- A pure LSUIElement=YES app struggles to give its settings/onboarding window keyboard focus and may show windows behind other apps; the reliable fix is the programmatic .accessory <-> .regular activation-policy dance plus NSApp.activate(ignoringOtherApps:).
- SMAppService.register() commonly returns success but lands in status .requiresApproval — the login item is OFF until the user toggles it in System Settings > General > Login Items. You MUST reflect status (not a stored bool) and guide the user, or the toggle will silently lie.
- Google KILLED custom URI schemes for the 'Desktop app' OAuth client type; that client requires loopback (127.0.0.1) redirect. You cannot do a pure ASWebAuthenticationSession custom-scheme flow with a Google Desktop client — you need the local loopback listener. This is the single biggest driver of the sandbox decision.
- Sandbox + loopback OAuth listener is a known friction point: it needs com.apple.security.network.server, and some loopback/network functionality that works in dev breaks once exported with Developer ID + sandbox, producing console sandbox-violation noise. Going unsandboxed for v0 sidesteps this entirely.
- Hardened runtime is mandatory for notarization (codesign --options runtime). It is independent of sandbox — you can (and here should) have hardened runtime WITHOUT sandbox.
- notarytool had intermittent submission timeouts / 'In Progress' hangs and stapler Error 65 reports in early 2026 (Apple-side). Build retry/backoff into CI and don't fail the pipeline on the first timeout.
- Don't store OAuth tokens or the SQLCipher key in UserDefaults or the GRDB file — Keychain only. Unsandboxed apps use the default login keychain; no keychain-access-groups entitlement is needed unless sharing across apps/extensions.
- Targeting macOS 26 Tahoe for Liquid Glass styling strands all non-Tahoe users; the shell itself needs nothing newer than macOS 13/14. Decouple the styling ambition from the deployment-target decision.

**Risks**
- [high] MenuBarExtra exists from macOS 13 Ventura and the .window style renders a rich popover panel while .menu drops images/custom button styles. — Apple MenuBarExtra docs + multiple corroborating tutorials (nilcoalescing, schurigeln) and the Tahoe demo repo all describe .menu vs .window behavior and the macOS 13 introduction.
- [high] SMAppService.mainApp is the correct, current launch-at-login mechanism (macOS 13+); register() can yield .requiresApproval and the UI must read .status, with a system-settings deep link for approval. — theevilbit SMAppService writeup enumerates the four status states (notRegistered/enabled/requiresApproval/notFound) and the approval flow; nilcoalescing 'Add launch at login setting' confirms reading status rather than a stored bool.
- [high] App Sandbox is optional for Developer ID (outside-MAS) distribution and mandatory only for the Mac App Store; hardened runtime is the notarization requirement and is independent of sandbox. — Apple App Sandbox docs state sandbox is required for MAS; Xojo/rsms notarization guides confirm Developer ID needs hardened runtime + notarization but not sandbox.
- [high] Google's 'Desktop app' OAuth client requires the loopback (127.0.0.1) redirect flow and no longer supports custom URI schemes, while loopback remains supported for desktop clients (deprecated only for iOS/Android/Chrome). — Google loopback-migration guide explicitly says loopback continues for desktop apps; Google native-app OAuth guide says custom URI schemes are no longer supported for that client type and recommends loopback/ephemeral local server.
- [medium] A sandboxed app needs com.apple.security.network.server to run a loopback OAuth listener, and this combination is fragile under Developer ID export. — App Sandbox docs map listening sockets to network.server (incoming connections); a Developer ID forum thread reports network functionality that works sandboxed-in-dev failing on Developer ID export with sandbox-violation messages. Exact loopback-listener-under-sandbox behavior on macOS 26 was not verified against a first-party doc — treat as a strong heuristic, not a guarantee.
- [high] notarytool + xcrun stapler (staple .app and DMG) is the supported 2026 notarization path; altool is retired. — rsms macOS distribution gist and 2024+ notarization guides describe notarytool submit --wait then stapler staple; Apple deprecated altool for notarization.
- [medium] macOS 14 Sonoma is a safe minimum deployment target for the shell with Xcode 26 / Swift 6.2; macOS 26 only needed for Tahoe Liquid Glass. — All required APIs (MenuBarExtra, SMAppService, ASWebAuthenticationSession) predate Sonoma; the choice is a product/install-base tradeoff not a technical blocker. Exact Xcode 26 minimum-target floor was not independently verified, but macOS 14 is well within range.
- [high] Stock MenuBarExtra(.window) cannot be programmatically dismissed after an in-panel action without third-party access to the NSStatusItem/NSWindow. — feedback-assistant report FB11984872 documents the missing API; orchetect/MenuBarExtraAccess exists specifically to work around it.

**Adversarial verifications**
- **[UNCERTAIN]** A sandboxed app needs com.apple.security.network.server to run a loopback OAuth listener, and this combination is fragile under Developer ID export.
  - Part 1 (network.server for a loopback listener) is WELL-SUPPORTED but not first-party-confirmed for the loopback case. Apple's entitlement reference (developer.apple.com/library/archive/.../EnablingAppSandbox.html) defines com.apple.security.network.server verbatim as "Network socket for listening for incoming connections initiated by other machines," and the modern entitlement page (developer.apple.com/documentation/bundleresources/entitlements/com.apple.security.network.server) calls it "A Boolean value indicating whether your app may listen for incoming network connections." A loopback OAuth listener bind()/listen()s a TCP server socket on 127.0.0.1, which is the listening-socket case that network.server governs. App Sandbox does NOT exempt TCP loopback listeners the way it exempts socketpair/UNIX-domain sockets (forum thread 126059, Quinn/eskimo: socketpair needs no entitlement; network.server was explicitly "not needed for local sockets" there — i.e. that thread is about UNIX sockets, not TCP). CAVEAT/refutation attempt: the doc says "initiated by OTHER machines," and a loopback callback is same-machine; I found NO first-party macOS-26 doc that explicitly states "a loopback TCP listener requires network.server." It is a strong, widely-followed heuristic (OAuth libs and the openidconnect Flutter package treat network.server + network.client as required for localhost callbacks), but not a doc-verified guarantee — exactly as the offered evidence concedes.

Part 2 (fragile under Developer ID export) is REAL AS A PATTERN but only INDIRECTLY tied to networking. Authoritative forum thread 808940 ("App Sandbox denies mach-register for Developer ID signed app but allows it for Apple Distribution signed app") proves App Sandbox behavior diverges between Developer ID and Apple Distribution signing for IDENTICAL entitlements — AMFI/certificate-restricted enforcement, with DTS engineer Quinn confirming and recommending app-group-prefixed names. That validates the dev-vs-export fragility class. HOWEVER that confirmed divergence is mach-register/Mach IPC, NOT network.server/loopback. I found no first-party source documenting that a network loopback listener specifically fails on Developer ID export; the "network worked in dev, failed on export" thread the evidence references could not be located/confirmed in current sources. So the network-listener-specific fragility is extrapolated from an IPC case, not directly evidenced.

NET: directionally correct and a sound engineering heuristic for the Daybrief native build (set network.server for the loopback listener; test the actual Developer-ID-exported, notarized build, not just the Xcode-run debug build — debug/profile builds auto-grant network.server, masking issues), but neither half is confirmed by a first-party macOS 26 doc at the loopback+network granularity. Verdict: uncertain, leaning plausible.
  - _caveat:_ Two unverified gaps: (1) No first-party doc explicitly states a same-machine loopback TCP listener requires network.server (the doc says "incoming connections initiated by other machines"); it is a strong heuristic, not a documented guarantee. (2) The only doc-confirmed Developer-ID-vs-distribution sandbox divergence I found is mach-register IPC (forum 808940), not network listening — the network-specific export fragility is extrapolated. Note also: network.server is auto-enabled for debug/profile builds, so a loopback listener can work when run from Xcode and only break in the exported Developer ID build — which is consistent with, and the likely real mechanism behind, the claimed fragility. Action: verify on the actual notarized Developer-ID-exported build and inspect Console for sandbox(network-bind) denials; do not trust the Xcode-run result.

## Scheduling & generate-on-wake without a daemon

**Summary.** No background execution beyond the running process is needed, and none is possible on macOS anyway: when the Mac sleeps the CPU is suspended and no app code runs, so any "fire at 7am" guarantee while asleep is impossible by design. The correct architecture is a self-rescheduling wall-clock alarm computed from the user's local fire-time, combined with a catch-up check on launch and on NSWorkspace.didWakeNotification that compares a persisted lastBriefDate against "today's fire-time has passed". Use a single DispatchSourceTimer (or Combine/Timer scheduled on the main run loop) to fire while running; do NOT use NSBackgroundActivityScheduler for the daily brief — it is a discretionary maintenance scheduler that is intentionally imprecise (default tolerance = half the interval) and will not honor a precise 7:00am user expectation. Guard the actual generation work with ProcessInfo.beginActivity(.userInitiated) so App Nap doesn't throttle the timer/network on this LSUIElement accessory app. The semantics ("if asleep at fire-time, generate on next wake/open") fall out naturally from the lastRun-vs-fireTime comparison, since the wake notification and app launch are the two moments the process regains CPU.


**Decisions**
- **Use a single recurring wall-clock alarm scheduled for the NEXT occurrence of the user's local fire-time, recomputed after every fire — not a fixed repeating interval.** — A 'daily at 7am' brief is a calendar/wall-clock event, not an N-second interval. Computing the next Date via Calendar.nextDate(after:matching:) and rescheduling on each fire keeps it aligned to local time across DST changes and avoids interval drift. An interval-based Timer drifts and mishandles DST/clock changes.
- **Implement the in-process timer with DispatchSourceTimer (or Timer on the main RunLoop) — NOT NSBackgroundActivityScheduler — for the user-facing daily fire.** — NSBackgroundActivityScheduler is documented as a discretionary maintenance scheduler whose default tolerance is HALF the interval and whose default QoS is background; it deliberately fires 'on average' near the deadline so the system can batch work. That is wrong for a brief the user expects at 7:00. DispatchSourceTimer/Timer fire at the requested wall-clock moment while the app is running. Reserve NSBackgroundActivityScheduler only if you later add truly discretionary maintenance (e.g. cache pruning).
- **Treat wake and launch as catch-up triggers, not as the primary schedule. On didWakeNotification and at app launch, run the same evaluateAndMaybeGenerate() routine that the timer calls.** — Because the CPU is suspended during sleep, a 7am fire that lands during sleep simply never happens. The first moment the process can act is wake or next open. Routing all three entry points (timer fire, wake, launch) through one idempotent evaluator gives 'generate on next wake/open' for free with no extra machinery.
- **Persist lastBriefDate (the calendar day a brief was successfully generated) in the GRDB/SQLCipher store and gate generation on: today's local fire-time has passed AND lastBriefDate != today.** — This single comparison implements all required semantics: fired-while-running (timer triggers, condition true), asleep-at-fire-time (wake triggers later, condition still true, generates once), and re-open same day (condition false, no duplicate). Storing only the day (not a timestamp) makes the 'once per day' guarantee trivial and timezone-stable. Persisting in the encrypted DB (not UserDefaults) keeps it consistent with the local-first, private-by-default principle and survives relaunch.
- **Wrap the generation work (connector fetches + LLM call) in ProcessInfo.processInfo.beginActivity(options: .userInitiated, reason:) ... endActivity().** — As an LSUIElement menu-bar accessory with no visible window, the app meets every App Nap criterion (not foreground, not updating a visible window, not audible, no power assertions). App Nap throttles timers and I/O for such processes. .userInitiated tells the system this is real work and exempts it from App Nap and deferral for the duration. Do NOT take an idle-sleep-disabling assertion — let the Mac sleep normally; honoring sleep is the whole point of generate-on-wake.
- **Register the wake observer on NSWorkspace.shared.notificationCenter (not NotificationCenter.default), and mark the scheduler/observer @MainActor under Swift 6 strict concurrency.** — didWakeNotification (and willSleepNotification) are posted ONLY on NSWorkspace's own notification center; registering on the default center silently receives nothing — a common, well-documented bug. AppKit notification delivery and any UI/menu-bar state mutation should be MainActor-isolated; the async generation can then hop to a background executor while the scheduling bookkeeping stays on the main actor, satisfying Swift 6.2 strict-concurrency checking without data-race warnings.

**Stack**
- `DispatchSourceTimer (Dispatch / libdispatch)` (macOS 26 SDK) — Primary in-process wall-clock alarm for the daily fire while the app runs. Precise, cancellable, reschedulable; no RunLoop-mode pitfalls. Alternative: Timer on RunLoop.main in .common mode.
- `NSWorkspace.didWakeNotification` (AppKit, macOS 26) — The wake trigger. Must observe via NSWorkspace.shared.notificationCenter. Pairs with willSleepNotification if you want to invalidate/teardown the running timer before sleep.
- `ProcessInfo.beginActivity(options:reason:) / endActivity:` (Foundation, macOS 26) — App Nap / energy-management opt-out scoped to the generation work. Use .userInitiated. Returns an opaque activity token (NSObjectProtocol) you must hold until endActivity.
- `Calendar.nextDate(after:matching:matchingPolicy:)` (Foundation, macOS 26) — Computes the next local occurrence of the user's fire-time (DateComponents hour/minute) — DST-safe wall-clock scheduling without manual date math.
- `GRDB.swift over SQLCipher` (locked project dependency) — Persists lastBriefDate and the user's configured fire-time in the encrypted store, keeping the run-state ledger local and private and surviving relaunch.
- `NSBackgroundActivityScheduler` (Foundation, macOS 26) — NOT for the daily brief. Optional, only for future discretionary maintenance (cache/log pruning) where imprecise timing is acceptable. Listed to document the explicit rejection.

**Patterns / concrete notes**
- Single source of truth evaluator: func evaluateAndMaybeGenerate(trigger: .timer | .wake | .launch). It (1) reads configured fireTime and lastBriefDate from GRDB, (2) computes todaysFireTime = Calendar.nextDate(...) rolled to today, (3) if Date() >= todaysFireTime && lastBriefDate != Calendar.current.startOfDay(for: Date) -> generate(); (4) on success, write lastBriefDate = today; (5) always call scheduleNextFire().
- scheduleNextFire(): let next = Calendar.current.nextDate(after: Date(), matching: DateComponents(hour: h, minute: m), matchingPolicy: .nextTime)! ; then arm a DispatchSourceTimer: timer.schedule(deadline: .now() + next.timeIntervalSinceNow); set its event handler to Task { @MainActor in await evaluateAndMaybeGenerate(trigger: .timer) }; timer.resume(). Recreate (do not reuse a one-shot) on each fire.
- Timer-vs-sleep fact to rely on: a Foundation Timer / DispatchSourceTimer scheduled for a deadline that elapses during sleep does NOT fire per missed interval — on a Mac that wakes it fires roughly once, late, on wall-clock catch-up. Do not depend on this fuzzy behavior; the didWakeNotification + lastRun check is the authoritative catch-up path. Treat any late timer fire as just another call into the idempotent evaluator (lastBriefDate guard makes double-fire harmless).
- Wake observer (MainActor): NSWorkspace.shared.notificationCenter.addObserver(forName: NSWorkspace.didWakeNotification, object: nil, queue: .main) { _ in Task { @MainActor in await self.evaluateAndMaybeGenerate(trigger: .wake) } }. Optionally also observe willSleepNotification to cancel the in-flight DispatchSourceTimer so it is freshly rearmed against the correct post-wake wall clock.
- App Nap scoping: let token = ProcessInfo.processInfo.beginActivity(options: .userInitiated, reason: "Generating daily brief"); defer { ProcessInfo.processInfo.endActivity(token) }. Wrap ONLY the fetch+synthesize span, not the whole app lifetime, so the Mac can still nap/sleep when idle.
- Idempotency / day key: store lastBriefDate as the ISO yyyy-MM-dd of the local day (or startOfDay Date). Comparison uses Calendar.current.isDate(_:inSameDayAs:). This makes 'once per calendar day' robust to manual regenerate, multiple wakes, and a late timer fire all landing the same day.
- Clock-change resilience: also recompute/reschedule on NSSystemClockDidChange (NSNotification.Name.NSSystemClockDidChange) and on NSSystemTimeZoneDidChange so a manual clock or timezone change re-aligns the 7am alarm. Cheap insurance for a wall-clock scheduler.
- Manual 'Generate now' menu action and the daily auto-fire share generate(); only the auto path writes lastBriefDate-gated logic, so a manual run mid-morning still updates lastBriefDate and suppresses a redundant auto-fire later that day.

**Gotchas**
- Registering the wake observer on NotificationCenter.default instead of NSWorkspace.shared.notificationCenter silently delivers nothing — didWakeNotification/willSleepNotification are posted only on NSWorkspace's notification center.
- NSBackgroundActivityScheduler default tolerance is HALF the interval and default QoS is background; it fires 'on average near the deadline' and can be deferred for energy/thermal reasons. Using it for the 7am brief would produce visibly wrong timing. It is a maintenance scheduler, not an alarm clock.
- No background execution exists while the Mac is asleep: the CPU is suspended (confirmed by Apple DTS). There is NO API in the core product to wake the machine to generate a brief, and attempting one (IOPMSchedulePowerEvent / scheduled wake) would violate the no-daemon / honor-sleep principle and require elevated/privileged setup.
- A repeating Timer with a fixed interval drifts and breaks across DST and manual clock changes. Always recompute the next absolute fire Date from DateComponents; never add 86400 seconds.
- Timer fires only in the RunLoop modes it's added to; if you ever use Timer instead of DispatchSourceTimer, add it to RunLoop.main in .common mode or it will pause while a menu/tracking loop is open. DispatchSourceTimer avoids this entirely.
- Holding a long-lived ProcessInfo activity (or any idle-sleep-disabling assertion) for the app's whole lifetime would prevent the Mac from sleeping and drain battery — exactly the opposite of the design. Scope beginActivity to the generation span only, and only with .userInitiated (not the idleSystemSleepDisabled options).
- Under Swift 6.2 strict concurrency, capturing self in the NSWorkspace observer closure and the DispatchSource event handler must be MainActor-isolated or you get data-race diagnostics; bridge into the async generation via Task { @MainActor in ... } and isolate the scheduler as @MainActor (or an actor with explicit hops).
- Sleep/wake notification delivery is documented as inconsistent on some MacBooks when on battery / with external displays. Do not treat didWakeNotification as guaranteed — the launch-time and any-foreground-open catch-up checks are the safety net that makes the feature reliable even if a wake notification is missed.
- If the app is fully quit at 7am (not just asleep), nothing fires — that's expected and acceptable for a local-first no-daemon app; generation happens on next launch via the same lastRun check. Set expectations in UI copy ('your brief is ready when you open Daybrief in the morning').

**Risks**
- [high] No background execution beyond the running process is needed or possible; while the Mac sleeps the CPU is suspended and no app code runs. — Apple Developer Forums thread 795461: Apple DTS (Quinn) states the CPU gets suspended on sleep, macOS can wake spontaneously and gives all processes execution time on wake, and there is no way to disable this. Forum thread 678161/127445 confirm it 'assumes your app is running' and won't run at the exact requested time.

- [high] ProcessInfo.beginActivity(options: .userInitiated) prevents App Nap and deferral for the scoped work; an LSUIElement accessory app with no visible window is otherwise an App Nap candidate. — Apple Energy Efficiency Guide 'App Nap' (candidate criteria: not foreground, no visible-window update, not audible, no assertions) and 'Prioritize Work at the App Level' which states NSActivityUserInitiated 'prevents the system from deferring the operations or putting your app in App Nap'.

## GRDB.swift + SQLCipher persistence (encrypted local-first SQLite for native macOS, 2026)

**Summary.** Use GRDB.swift v7.11.0 (June 2026) as the persistence layer. As of GRDB 7.10.0 (Feb 2026), SQLCipher is supported over SPM via the official Zetetic package `sqlcipher/SQLCipher.swift` (>= 4.11.0), but it still requires forking GRDB and uncommenting four marked lines in `Package.swift` (the "GRDB+SQLCipher" comments) — there is no clean trait/product to flip yet, which is the single biggest setup gotcha. On Apple Silicon macOS this path is clean: SQLCipher.swift uses Apple's CommonCrypto as its crypto backend (no OpenSSL/libcrypto, no extra signing/hardened-runtime exceptions). For Daybrief's design, generate a random 256-bit key on first launch, store it in the Keychain (raw bytes, `kSecAttrAccessibleAfterFirstUnlock`), and apply it as a raw key (`PRAGMA key = "x'<64-hex>'"`) inside `Configuration.prepareDatabase` to skip KDF; use `DatabaseQueue` (menu-bar app, single writer, modest data) with `DatabaseMigrator`. GRDB 7 is built in Swift 6 language mode with full Sendable conformance and a MainActor-first `ValueObservation`, integrating cleanly with `@Observable` view models.


**Decisions**
- **GRDB.swift 7.11.0 via SPM, forked to enable SQLCipher (uncomment the four 'GRDB+SQLCipher' lines in Package.swift)** — GRDB 7.10.0+ ships official SQLCipher-over-SPM support but Package.swift cannot yet express it as a selectable product/trait, so the maintainer's documented path is to fork and edit. The fork adds dependency on sqlcipher/SQLCipher.swift (>=4.11.0) and defines SQLITE_HAS_CODEC (cSettings) + SQLITE_HAS_CODEC and SQLCipher (swiftSettings). Verified against the live master Package.swift.
- **Crypto backend = CommonCrypto (default on Apple platforms via SQLCipher.swift)** — On macOS/iOS/visionOS the official SQLCipher.swift uses Apple's CommonCrypto framework — no libcrypto/OpenSSL dependency, no extra dylibs to sign or notarize. This keeps the hardened-runtime / notarization story trivial on Apple Silicon. OpenSSL is only the Linux backend.
- **DatabaseQueue, not DatabasePool** — Daybrief is a single-process menu-bar app with a single writer and small data volume; DatabaseQueue serializes all access and keeps one long-lived connection, which also sidesteps the DatabasePool passphrase-availability failure mode (pool opens connections on demand and can fail mid-session if the Keychain key becomes unavailable). DatabaseQueue keeps its connection alive even if the key later turns unavailable.
- **Random 256-bit raw key (not a user passphrase), stored in Keychain, applied as PRAGMA key = x'<hex>'** — Local-first/private-by-default with no user-entered password: generate 32 random bytes via SecRandomCopyBytes on first launch, hex-encode to 64 chars, and pass as a raw key so SQLCipher skips PBKDF2 key derivation (faster open, and the entropy is already full). Store raw bytes in Keychain item; load only inside prepareDatabase.
- **kSecAttrAccessibleAfterFirstUnlock for the Keychain key item** — Balances security with a menu-bar app that may run/refresh in the background after the user logs in. AfterFirstUnlock keeps the key reachable for background brief assembly while still protecting it before first unlock. (AfterFirstUnlockThisDeviceOnly if no iCloud Keychain sync is desired — recommended for a device-bound local DB.)
- **DatabaseMigrator for schema, with eraseDatabaseOnSchemaChange only in DEBUG** — GRDB's first-class migration story: each migration runs in its own transaction with deferred FK checks, applied-migration memory is stored in the DB itself, and migrate(_:) only runs non-applied migrations. eraseDatabaseOnSchemaChange speeds dev iteration but must never ship enabled (it wipes user data).
- **Drive @Observable view models from ValueObservation (optionally via GRDBQuery)** — GRDB 7's ValueObservation is MainActor-first: start() and callbacks are MainActor-isolated by default, so you can assign fetched values straight to an @Observable model's properties without MainActor.assumeIsolated. GRDBQuery (groue/GRDBQuery) is the official SwiftUI companion if you want @Query-style property wrappers; for a menu-bar app a hand-rolled @Observable + observation.start(in:) is simplest.
- **Optional FTS5 virtual table for chat-context search over brief items** — FTS5 is enabled by GRDB's default swiftSettings (.define("SQLITE_ENABLE_FTS5")) and survives the SQLCipher fork. Use db.create(virtualTable:using:FTS5()), ideally an external-content table referencing the brief-items table to avoid duplicating text, with porter/unicode61 tokenizer. The whole FTS index is encrypted because it lives in the same SQLCipher file.

**Stack**
- `GRDB.swift` (7.11.0 (released 2026-06-01; SQLCipher+SPM support landed in 7.10.0, Feb 2026)) — SQLite toolkit: connections, migrations, query interface, ValueObservation, FTS5. Min Swift 6.1+/Xcode 16.3+, macOS 10.15+ — well within Daybrief's macOS 26/Xcode 26 target.
- `sqlcipher/SQLCipher.swift` (>= 4.11.0 (SQLCipher 4.x core; v3.4+ minimum)) — Official Zetetic SPM package referenced verbatim by GRDB's Package.swift; provides the SQLCipher-enabled libsqlite3 with CommonCrypto backend on Apple platforms. 256-bit AES.
- `Security.framework (Keychain + SecRandomCopyBytes)` (macOS 26 system) — Generate the random DB key (SecRandomCopyBytes) and persist it (SecItem* APIs) with kSecAttrAccessible protection; nothing leaves the machine.
- `GRDBQuery (optional)` (current (June 2026)) — Official SwiftUI companion for GRDB if @Query-style declarative DB binding in views is wanted; otherwise plain @Observable + ValueObservation suffices.
- `Observation framework / @Observable` (macOS 14+ (built into macOS 26)) — Modern view-model layer that ValueObservation feeds on the MainActor.

**Patterns / concrete notes**
- SPM SQLCipher enablement (fork groue/GRDB.swift, edit Package.swift — VERBATIM from live master): uncomment (1) dependencies.append(.package(url: "https://github.com/sqlcipher/SQLCipher.swift.git", from: "4.11.0")); (2) cSettings.append(.define("SQLITE_HAS_CODEC")); (3) swiftSettings.append(.define("SQLITE_HAS_CODEC")); (4) swiftSettings.append(.define("SQLCipher")). Then DELETE the GRDBSQLite library + systemLibrary target, UNCOMMENT the GRDBSQLCipher target (depends on .product(name: "SQLCipher", package: "SQLCipher.swift")), and in the GRDB target DELETE the GRDBSQLite dependency and UNCOMMENT both .product(name: "SQLCipher", package: "SQLCipher.swift") and .target(name: "GRDBSQLCipher").
- Open encrypted DB with PASSPHRASE (GRDB convenience): var config = Configuration(); config.prepareDatabase { db in try db.usePassphrase("secret") }; let dbQueue = try DatabaseQueue(path: dbPath, configuration: config).
- Open with RANDOM RAW KEY (recommended for Daybrief — skips KDF): pass the SQLCipher raw-key blob literal as the passphrase string: try db.usePassphrase("x'2DD29CA8...A8086D99'") where the hex is exactly 64 chars = 32 bytes = 256 bits. (SQLCipher converts x'...' directly to key bytes with no derivation.)
- First-launch key generation: var bytes = [UInt8](repeating: 0, count: 32); SecRandomCopyBytes(kSecRandomDefault, 32, &bytes); hex-encode; store raw 32 bytes in Keychain (SecItemAdd, kSecClassGenericPassword, kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly). On later launches SecItemCopyMatching to retrieve.
- Load key lazily inside prepareDatabase (GRDB-recommended for both lifetime AND availability): var config = Configuration(); config.prepareDatabase { db in let key = try loadKeyFromKeychain(); try db.usePassphrase(key) } — load INSIDE the closure, never capture it in config, so it isn't held in memory and so connection creation fails cleanly if the Keychain key is unavailable.
- Extra SQLCipher PRAGMAs go in prepareDatabase AFTER usePassphrase, e.g. try db.execute(sql: "PRAGMA cipher_page_size = 4096") / "PRAGMA kdf_iter = ..." — but with a raw key, KDF iters are irrelevant. Use SQLCipher 4 defaults; no cipher_compatibility needed for a fresh DB.
- Migrations: var migrator = DatabaseMigrator(); migrator.registerMigration("v1") { db in try db.create(table:"briefItem"){ t in t.autoIncrementedPrimaryKey("id"); t.column("source", .text); t.column("title", .text); t.column("body", .text); t.column("createdAt", .datetime) } }; then try migrator.migrate(dbQueue). Add #if DEBUG migrator.eraseDatabaseOnSchemaChange = true #endif.
- Swift 6 records must be Sendable: model brief items / accounts as structs (not classes / not GRDB's deprecated Record class) conforming to Codable + FetchableRecord + PersistableRecord; use computed (var) databaseSelection, never stored let, to avoid 'not concurrency-safe' errors.
- ValueObservation -> @Observable on MainActor (GRDB 7): @MainActor func observeBrief() { let obs = ValueObservation.tracking { db in try BriefItem.order(Column("createdAt").desc).fetchAll(db) }; cancellable = obs.start(in: dbQueue, onError: { ... }) { items in self.items = items } } — the onChange closure is already MainActor-isolated, no assumeIsolated needed.
- Async sequence form: for try await items in observation.values(in: dbQueue) { ... } — NOTE in GRDB 7 .values iterates on the cooperative thread pool by default; pass scheduling: .mainActor if you need main-thread delivery.
- Async read/write off the cooperative pool: let n = try await dbQueue.read { db in try BriefItem.fetchCount(db) }; try await dbQueue.write { db in try item.insert(db) } — prefer await in Task contexts to avoid blocking; sync dbQueue.read/write is fine for tiny fast queries even on main.
- FTS5 for chat-context search (external-content, encrypted with the rest): migrator.registerMigration("fts") { db in try db.create(virtualTable: "briefItem_ft", using: FTS5()) { t in t.synchronize(withTable: "briefItem"); t.column("title"); t.column("body"); t.tokenizer = .porter(wrapping: .unicode61()) } } then query with let pattern = FTS5Pattern(matchingAllTokensIn: query); BriefItem.matching(pattern).fetchAll(db).
- Verify encryption on disk (CI/manual check): the first 16 bytes are a random salt, so `head -c 16 daybrief.sqlite | xxd` must NOT show the ASCII 'SQLite format 3\000' magic; an UNencrypted DB starts with that string, an SQLCipher DB shows random bytes. Add a test that opens the file path WITHOUT the key and asserts it throws 'file is encrypted or is not a database' (SQLite error 26).
- Raw-byte key handling for hardened lifetime (optional): use Data + defer { data.resetBytes(in: 0..<data.count) }, or drop to sqlite3_key(db.sqliteConnection, ...) directly inside prepareDatabase for full control over passphrase byte lifetime.

**Gotchas**
- No turnkey SQLCipher product: as of GRDB 7.11.0 you MUST fork GRDB and hand-edit Package.swift (uncomment the four 'GRDB+SQLCipher' lines + swap GRDBSQLite->GRDBSQLCipher target). There is no SPM trait/product to select yet; Xcode lacks the package-traits support the maintainer wants. Pin your fork and document the diff, because any GRDB upgrade requires re-applying it. Downstream packages (GRDBQuery, SQLiteData) would also need to point at the fork.
- 'GRDB.swift/SQLCipher' (CocoaPods) and the SQLCipher SPM package must be the ONLY SQLite in the build. Linking both system SQLite and SQLCipher causes linker/runtime symbol conflicts. Don't also pull in a vanilla GRDB.swift dependency anywhere in the graph.
- Providing a key does NOT encrypt an existing plaintext DB — SQLCipher throws 'SQLite error 26: file is encrypted or is not a database'. To encrypt existing data you must ATTACH a new encrypted DB and SELECT sqlcipher_export('encrypted'). For Daybrief (greenfield) just create encrypted from day one.
- DatabasePool + Keychain key = unpredictable failures: a pool opens connections on demand, so if the Keychain key becomes unavailable (e.g. wrong kSecAttrAccessible class while screen locked), some reads/writes succeed on existing connections and others fail when a new connection is opened. DatabaseQueue avoids this (one long-lived connection). This is a documented GRDB caveat.
- Load the key INSIDE prepareDatabase, not into the Configuration up front: capturing it means it lives as long as the config/connection AND it breaks across a future key change (the closure would keep re-applying the stale captured key). GRDB's own docs flag the captured-passphrase bug.
- Changing the key on a DatabasePool requires barrierWriteWithoutTransaction { db in try db.changePassphrase(...); dbPool.invalidateReadOnlyConnections() } and does NOT protect open snapshots — they can throw mid-change. Mostly avoided by using DatabaseQueue.
- Swift 6 strict concurrency will flag every non-Sendable record: classes (especially GRDB's deprecated Record base class) and stored `static let databaseSelection` both error. Use structs and `static var` computed selection. GRDB 7 itself is fine (built in .v6 language mode, Sendable conformances added), the friction is in YOUR model types.
- GRDB 7 changed ValueObservation.values default scheduler to the cooperative thread pool (was main). If you were relying on main-thread delivery for UI you must pass scheduling: .mainActor or you'll mutate @Observable state off the main actor.
- String passphrases (and even the x'hex' raw-key string) are NOT guaranteed to be scrubbed from memory — Swift String byte lifetime is uncontrolled. For a security-sensitive product, prefer Data + resetBytes or sqlite3_key directly. For a local-first single-user app this is usually acceptable, but record it as a known limitation.
- SQLCipher salt-in-header means you cannot rely on a fixed file magic — encryption-verification tests must check the ABSENCE of 'SQLite format 3' rather than the presence of a specific byte pattern (the salt is random per database).
- SQLCipher 3 vs 4 on-disk format differ; opening an old SQLCipher-3 file with SQLCipher 4 needs PRAGMA cipher_compatibility = 3 (or a migration). Not an issue for a fresh DB, but pin the SQLCipher major version so a future bump doesn't silently break existing user DBs.
- Min platform floor is fine but watch SQLite feature gating: UPSERT and RETURNING need SQLite 3.35+; SQLCipher 4.x bundles a recent SQLite so these ARE available (the system-SQLite macOS-version caveats in GRDB docs don't apply once you're on SQLCipher's bundled SQLite). Confirm the bundled SQLite version of the SQLCipher.swift release you pin.

DatabaseWriter.swift: `public protocol DatabaseWriter: DatabaseReader` (inherits the Sendable requirement). (5) PR #1510 "Add Sendable conformances and unavailabilities" (merged 2024-03-17) is the work that introduced these conformances, matching the cited search result. So the connection types and reader/writer protocols are concretely Sendable via explicit declarations — not merely inferred from the language mode.
  - _caveat:_ The blanket phrasing "all core types are Sendable" overreaches and was correctly flagged as inferred rather than line-verified by the evidence-offerer. ## Keychain secrets management (Security framework, macOS data protection keychain) for Daybrief native rebuild

**Summary.** Store all secrets (per-connection OAuth access+refresh tokens, BYO OAuth client id/secret for Google+Slack, the LLM API key, and the SQLCipher DB key) as kSecClassGenericPassword items in the macOS DATA PROTECTION keychain — opt in explicitly with kSecUseDataProtectionKeychain on every SecItem call, never the legacy file-based keychain. Use kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly so a scheduled brief can be generated after wake even while the screen is locked, while keeping secrets off iCloud and out of Time Machine/migration; the only exception is the SQLCipher DB key, which can stay AfterFirstUnlockThisDeviceOnly to match (the DB file is local). For a single standalone sandboxed app you do NOT need the keychain-access-groups entitlement — the app gets an implicit access group from its application-identifier (TeamID.bundleID); only add keychain-access-groups if a helper/XPC/login-item must share items. Wrap SecItem in a thin Swift actor with an upsert (add-then-update-on-errSecDuplicateItem) helper, key items by a stable service+account naming scheme, and guarantee secrets never reach os_log by marking all interpolations .private/.sensitive (never .public).


**Decisions**
- **Use the data protection keychain (pass kSecUseDataProtectionKeychain: true on EVERY SecItem call), not the legacy file-based keychain** — Per Apple TN3137 and DTS (Quinn) guidance, SecItem talks to the data protection keychain only if you supply kSecUseDataProtectionKeychain or kSecAttrSynchronizable; otherwise it falls through a limited shim to the deprecated file-based keychain. The data protection keychain is an iOS-style SQLite store that supports kSecAttrAccessible classes, SecAccessControl, Secure Enclave, and predictable cross-platform behavior. It is available to any app running in a user login context (Daybrief is a normal menu-bar app, so this is satisfied).
- **kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly for all items (OAuth tokens, client id/secret, LLM key, DB key)** — A menu-bar app that must generate a brief shortly after wake may run while the screen is locked. WhenUnlocked items become unreadable whenever the screen is locked and reads fail with errSecInteractionNotAllowed (-25308), which would break scheduled/post-wake brief generation. AfterFirstUnlock keeps items readable after the first post-boot login for the rest of the session, including while the screen is locked — exactly the background-refresh use case Apple documents. The ThisDeviceOnly suffix keeps the secrets off iCloud Keychain and excludes them from device migration/backup, satisfying the local-first/private-by-default principle. The tradeoff (slightly larger window where an unlocked-once-then-locked machine has decryptable secrets at rest) is acceptable and necessary for the feature; WhenUnlockedThisDeviceOnly would be strictly more private but would intermittently break the core scheduled-brief feature.
- **No keychain-access-groups entitlement for the v0 single-app build; rely on the implicit application-identifier access group** — Apple docs/forums: keychain-access-groups is only required to SHARE items between apps/extensions. A standalone app automatically gets an access group equal to its application-identifier (TeamID.bundleID) and can read/write its own items without the entitlement. Adding an empty or unnecessary keychain-access-groups list under App Sandbox is a common source of errSecMissingEntitlement (-34018). Add the entitlement only if/when a separate helper, XPC service, or login-item LaunchAgent needs to read the same secrets (e.g. a background brief generator), and then list a shared group like $(AppIdentifierPrefix)co.daybrief.shared on both.
- **Thin Swift wrapper as an actor with an explicit upsert and typed errors** — Generic-password uniqueness is (kSecClass, kSecAttrService, kSecAttrAccount, kSecAttrAccessGroup, kSecAttrSynchronizable). Naive SecItemAdd on an existing key returns errSecDuplicateItem. A single store(_:for:) that does SecItemAdd then, on -25299, SecItemUpdate (or delete-then-add) prevents the most common keychain bug. An actor gives Swift 6 strict-concurrency-safe serialized access without locking the SecItem C API yourself, and a typed KeychainError mapping SecCopyErrorMessageString keeps call sites clean.
- **Item-naming scheme: kSecAttrService = reverse-DNS namespace per secret category; kSecAttrAccount = the connection/instance identifier** — Service+account is the primary key, so encoding category in service and instance in account makes items collision-free, enumerable, and self-documenting. It cleanly supports Gmail multi-account (multiple accounts under the same connector service) and per-connection token rotation, and makes 'delete all secrets for connection X' a single deterministic query.
- **Never log secret values; route all logging through OSLog with .private/.sensitive interpolation and log only opaque references** — By default os_log redacts dynamic strings, but developers frequently make values .public for debugging and leak them. Mandate that no secret is ever interpolated; log only the item identifier (service/account) and operation outcome. Use OSLogPrivacy.sensitive (always redacted, even in DEBUG) or .private with .mask(.hash) if a stable correlatable token is needed. Add a lightweight test/lint guard that fails if a secret type is passed to a log call.

**Stack**
- `Security.framework (SecItem* C API)` (macOS 26 Tahoe SDK (Xcode 26)) — Canonical, dependency-free keychain access. SecItemAdd / SecItemCopyMatching / SecItemUpdate / SecItemDelete with kSecClassGenericPassword. Wrap directly rather than taking a third-party dependency for a security-critical, small surface.
- `os.Logger (OSLog / unified logging)` (macOS 26) — Privacy-aware logging with compile-time .private/.sensitive redaction so secrets cannot leak into the unified log even by accident.
- `CryptoKit (optional, supporting)` (macOS 26) — If a random SQLCipher DB key must be generated on first launch, use SymmetricKey(size: .bits256) / SecRandomCopyBytes for the key material before storing it in the keychain. Not strictly part of keychain access but adjacent to DB-key provisioning.

**Patterns / concrete notes**
- Always pass kSecUseDataProtectionKeychain: kCFBooleanTrue (Swift: true as CFBoolean) in EVERY query dict — add, copy, update, delete — so all operations hit the same (data protection) keychain. Inconsistency between add and read is a classic 'item not found' bug.
- Generic-password primary key = (kSecClass, kSecAttrService, kSecAttrAccount, kSecAttrAccessGroup, kSecAttrSynchronizable). Set kSecAttrSynchronizable: false explicitly to keep items local and to make the key deterministic.
- Upsert pattern: build base query (class+service+account+accessGroup), call SecItemAdd with kSecValueData + kSecAttrAccessible; if status == errSecDuplicateItem (-25299) call SecItemUpdate(baseQuery, [kSecValueData: newData]). Keep accessibility on the ADD only (you cannot change kSecAttrAccessible via update reliably; delete+add to migrate accessibility).
- Read pattern: query with kSecReturnData: true, kSecMatchLimit: kSecMatchLimitOne; handle errSecItemNotFound (-25300) as nil, errSecInteractionNotAllowed (-25308) as 'locked, retry after unlock'.
- Store the value as Data: tokens/keys as UTF-8 Data; the SQLCipher key as raw 32-byte Data (or hex). GRDB/SQLCipher PRAGMA key is supplied from this Data at DB open time — never persist the key anywhere but the keychain.
- Item-naming scheme — kSecAttrService (reverse-DNS, per category): 'co.daybrief.oauth.google' / 'co.daybrief.oauth.slack' (token blobs), 'co.daybrief.oauthclient.google' / 'co.daybrief.oauthclient.slack' (BYO client id+secret), 'co.daybrief.llm' (model API key), 'co.daybrief.db' (SQLCipher key). kSecAttrAccount = the instance: per-Gmail-account address or a stable connectionID UUID for tokens; 'default' for the singletons (db key, llm key); 'client' for the client-credential record. Store access_token + refresh_token + expiry together as one JSON/Codable blob per connection account rather than separate items, so refresh is atomic.
- kSecAttrAccessible: kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly on every item. Do NOT rely on a default — when kSecAttrAccessible is omitted the effective default is WhenUnlocked-class, which breaks locked-screen reads.
- kSecAttrLabel: human-readable label (e.g. 'Daybrief — Google (alice@gmail.com)') so items are identifiable in Keychain Access.app during debugging; never put secret material in the label.
- Entitlements (v0 single app, sandboxed): com.apple.security.app-sandbox = true; NO keychain-access-groups key. The app's implicit access group = $(AppIdentifierPrefix)<bundleID> from the application-identifier. If a helper/XPC is added later, add keychain-access-groups: ['$(AppIdentifierPrefix)co.daybrief.shared'] to BOTH targets and set kSecAttrAccessGroup explicitly on shared items.
- Concurrency: expose the wrapper as `actor KeychainStore` (Swift 6 strict concurrency). SecItem APIs are thread-safe but the actor serializes Daybrief's own add/update races (e.g. token refresh from two connectors).
- Rotation: on OAuth refresh, overwrite the per-connection token blob via SecItemUpdate (same service+account) — do not accumulate items. On 'disconnect connector', SecItemDelete the matching service+account. Provide a 'wipe all secrets' that enumerates by service prefix and deletes (e.g. for a full reset / uninstall hygiene). For the LLM key and client secrets, rotation = user pastes a new value → upsert overwrites.
- Logging: `logger.debug("keychain store \(service, privacy: .public)/\(account, privacy: .public) ok")` is fine; the value is NEVER interpolated. If a correlatable handle is needed use `\(token, privacy: .private(mask: .hash))`. Prefer `.sensitive` for anything that could be secret-adjacent.

**Gotchas**
- Omitting kSecUseDataProtectionKeychain on macOS silently routes you to the legacy file-based keychain, which has different uniqueness/attribute semantics and does NOT honor kSecAttrAccessible the same way. Mixed usage (add to one keychain, read from the other) produces phantom errSecItemNotFound. Set it everywhere or nowhere — set it everywhere.
- If kSecAttrAccessible is not specified, the item defaults to a WhenUnlocked-class protection. For a menu-bar app that wakes and generates a brief while the screen is locked, that default causes intermittent errSecInteractionNotAllowed (-25308). Always set AfterFirstUnlockThisDeviceOnly explicitly.
- Even with AfterFirstUnlock, reads can still fail right at boot before the first unlock (-25308) and occasionally in edge background contexts. The brief generator must handle 'keychain locked' gracefully (defer until unlock / surface a 'waiting for unlock' state) rather than crashing or treating it as 'no credentials'.
- errSecMissingEntitlement (-34018): on macOS this only occurs against the data protection keychain. The most common self-inflicted cause is adding a keychain-access-groups entitlement that lists a group the app isn't actually entitled to, or adding it unnecessarily and then setting kSecAttrAccessGroup to something outside the implicit group. For a single app, leave keychain-access-groups out entirely and do not set kSecAttrAccessGroup.
- kSecAttrAccessible CANNOT be changed by SecItemUpdate in a fully reliable way — if you ever need to migrate an item's accessibility class, delete and re-add it. Plan the accessibility class correctly up front.
- The data protection keychain requires a user login context. It works for a normal menu-bar app/LSUIElement agent, but a true launchd DAEMON (root, pre-login) cannot use it and must use the file-based keychain or System keychain — relevant only if Daybrief ever adds a privileged background daemon (it shouldn't, given the no-24/7-server principle).
- Don't split one connection's access_token, refresh_token, and expiry across three keychain items — a refresh that updates only some leaves an inconsistent state. Store one Codable blob per (connector, account).
- Mac sandbox + keychain: enabling App Sandbox does NOT by itself require keychain-access-groups for the app's own items; the implicit application-identifier group still works under sandbox. The entitlement is only the SHARING mechanism.
- OSLog redacts dynamic strings by default, but the moment a developer marks a value .public for debugging and forgets to revert, the secret persists to the unified log (readable via Console.app / log collect / sysdiagnose). Treat any .public on a secret-bearing call site as a release blocker.
- iCloud Keychain accidental sync: if you ever set kSecAttrSynchronizable: true (or use a non-ThisDeviceOnly accessibility class with sync enabled), secrets leave the machine — directly violating the local-first principle. Pin synchronizable=false and use ThisDeviceOnly classes.

**Risks**
- [high] On macOS, kSecAttrAccessibleAfterFirstUnlock(ThisDeviceOnly) keeps items readable while the screen is locked (after the first post-boot login), enabling post-wake brief generation; WhenUnlocked items fail with -25308 when the screen is locked. — Apple documentation (kSecAttrAccessibleAfterFirstUnlock / kSecAttrAccessibleWhenUnlocked) and multiple DTS forum threads describe AfterFirstUnlock as the class for background-refresh access and WhenUnlocked as readable only while unlocked; -25308 (errSecInteractionNotAllowed) reports confirm locked-state read failures. macOS 'unlocked' tracks the login/screen-lock session for the data protection keychain.
- [high] A single standalone sandboxed macOS app does NOT need the keychain-access-groups entitlement and gets an implicit access group from its application-identifier (TeamID.bundleID). — Apple keychain-access-groups entitlement docs ('not required for a single app; only for sharing') plus DTS forum guidance that apps default to an access group equal to the application-identifier and that -34018 stems from listing groups you aren't entitled to. The com.apple.token nuance on macOS 14 is separate (implicit on macOS, occasionally required) and does not affect generic-password items in the app's own group.
- [high] Generic-password uniqueness is determined by (kSecClass, kSecAttrService, kSecAttrAccount, kSecAttrAccessGroup, kSecAttrSynchronizable), so the add-then-update-on-errSecDuplicateItem upsert is the correct pattern. — Apple DTS 'SecItem: Fundamentals' (Quinn) thread enumerates exactly these primary-key attributes and recommends the add/update pattern; corroborated by errSecDuplicateItem documentation.
- [medium] When kSecAttrAccessible is omitted, the effective default protection is a WhenUnlocked-class accessibility, so it must be set explicitly for a locked-screen-capable app. — Widely reported (react-native-keychain issue #478 and multiple write-ups) that the default is WhenUnlocked-class; Apple's own per-constant pages render via JS and could not be body-scraped to get a single canonical sentence, but the behavior is consistent across secondary sources and matches observed -25308 failures. Mitigated regardless because the recommendation is to ALWAYS set the attribute explicitly.
- [high] The data protection keychain is available to Daybrief because it runs as a normal user-context menu-bar app (not a pre-login daemon). — TN3137 / DTS: data protection keychain is available to apps and app extensions running in a user context; only launchd daemons running outside a user context are excluded. A menu-bar LSUIElement app runs in the user's login session.
- [high] OSLog default redaction plus enforced .private/.sensitive interpolation prevents secrets from reaching the unified log. — Apple unified-logging docs and multiple Swift logging guides: dynamic strings are redacted by default; .public is required to expose them. Risk is human (marking .public), not technical, which is why an enforced convention/test guard is recommended.

**Adversarial verifications**
- **[UNCERTAIN]** When kSecAttrAccessible is omitted, the effective default protection is a WhenUnlocked-class accessibility, so it must be set explicitly for a locked-screen-capable app.
  - Split outcome: the operative conclusion (set kSecAttrAccessible explicitly for locked-screen/background use) is CONFIRMED and Apple-recommended; the "default = WhenUnlocked" premise is well-corroborated by SECONDARY sources but NOT confirmable from a current Apple PRIMARY source — hence uncertain overall.

The canonical Apple SecItem.h header (Apple-FOSS-Mirror/Security, keychain/SecItem.h) documents kSecAttrAccessible only as "Specifies a dictionary key whose value indicates when your application needs access to an item's data" and does NOT name any default.   - _caveat:_ Could not verify the "WhenUnlocked is the default" premise from a current Apple primary source: SecItem.h header explicitly names no default and the per-constant pages are JS-rendered. Premise rests on secondary sources (Fortify, write-ups, forum framing). ## Google OAuth for an OSS native macOS desktop app (Calendar + Gmail), bring-your-own OAuth client

**Summary.** The bring-your-own (BYO) Google OAuth client per user is the correct and necessary architecture: gmail.readonly is a Google RESTRICTED scope, and critically gmail.metadata is ALSO restricted (not a non-restricted escape hatch — this contradicts the dimension's premise). A single shared OSS client cannot serve unverified restricted scopes to >100 users and would need brand verification + restricted-scope verification (and CASA only if data touches a server). Implement the OAuth 2.0 installed-app flow with a loopback redirect (http://127.0.0.1:<ephemeral-port>) + PKCE (S256). On macOS the loopback redirect MUST be caught by a local HTTP listener (Network framework or AppAuth's OIDRedirectHTTPHandler); ASWebAuthenticationSession alone cannot catch an http loopback redirect — it only matches a custom callbackURLScheme — so it serves only as the in-app browser presenter, not the redirect receiver. The hard operational reality for a daily-brief app: while each user's own consent screen stays in "Testing" status, Google revokes refresh tokens after 7 days (verified current), which silently breaks a daily app weekly; the design must either guide each user to publish their own app to "In production" (no Google review needed because they are the sole/internal user of their own client) or build robust 7-day re-consent handling.


**Decisions**
- **Use the BYO per-user Google OAuth Desktop client; do not ship a shared OSS client** — gmail.readonly AND gmail.metadata are both RESTRICTED scopes. A shared client serving restricted scopes to the public requires brand verification + restricted-scope verification (demo video, justification) and, if restricted data ever transits a server, a paid CASA security assessment. An OSS project cannot realistically pass/maintain this for arbitrary downstream users. BYO client makes each user the owner+sole user of their own client, sidestepping public verification entirely.
- **Installed-app flow with loopback redirect (http://127.0.0.1:<random-port>/) + PKCE S256, no client secret reliance** — Google explicitly continues to support the loopback IP flow for the Desktop app client type (deprecated only for iOS/Android/Chrome, no desktop sunset date). Custom-scheme and OOB copy/paste redirects are no longer supported for desktop. PKCE (code_challenge_method=S256) is the recommended hardening; the Desktop client 'secret' is not a true secret for a public client, so PKCE is what actually protects the code exchange.
- **Catch the loopback redirect with a local HTTP listener (Network framework NWListener on 127.0.0.1, ephemeral port), present the consent UI via ASWebAuthenticationSession** — ASWebAuthenticationSession's callbackURLScheme matches a CUSTOM scheme only and cannot receive an http://127.0.0.1 loopback redirect (its https-callback support is for associated-domain Universal Links, not arbitrary loopback). The canonical native pattern (per AppAuth-iOS OIDRedirectHTTPHandler) is: start a loopback HTTP server on a random free port, derive the redirect_uri from it, open the system browser/auth session for consent, and let the server capture the ?code= callback. ASWebAuthenticationSession is still useful as the browser surface (gives the user a real signed-in Google session and a trusted UI), but it is not the redirect receiver.
- **Request the minimum scopes: calendar.events.readonly (or calendar.readonly) + gmail.metadata if possible, else gmail.readonly** — Calendar minimum read scope is https://www.googleapis.com/auth/calendar.events.readonly (sensitive, not restricted). For Gmail, gmail.metadata (From/Subject/Date headers + labels + timestamps, no body) is sufficient for a brief that surfaces what arrived without quoting content, and has a smaller data footprint — BUT it is still a RESTRICTED scope, so it does NOT reduce the consent-screen status / verification class versus gmail.readonly. Choose gmail.metadata for privacy-minimization, not to avoid restricted-scope friction (it doesn't).
- **Guide each user to publish THEIR OWN consent screen to 'In production' to escape the 7-day refresh-token expiry; build 7-day re-consent fallback regardless** — A Testing-status app revokes refresh tokens 7 days after consent (verified current). A daily-brief app would break weekly. Because the user owns their own client and is its only user, they can set publishing status to 'In production' without passing Google review (review/verification is only enforced when the unverified-app warning would be shown to OTHER users / >100 users). The app should also detect invalid_grant ('Token has been expired or revoked') and trigger silent re-auth as a safety net.

**Stack**
- `AuthenticationServices (ASWebAuthenticationSession)` (macOS 26 SDK (Xcode 26)) — Presents Google's consent/sign-in in a trusted system browser surface; gives the user their real Google session. Used as the external user-agent ONLY — it cannot catch the http loopback redirect itself.
- `Network framework (NWListener) for loopback HTTP server` (macOS 26) — Spins up an ephemeral 127.0.0.1:<port> listener to receive the OAuth ?code= redirect — the part ASWebAuthenticationSession cannot handle. Pure first-party, Swift 6 concurrency-friendly, no third-party dep.
- `AppAuth-iOS (optional reference/library)` (1.7.x) — Reference implementation of the exact native pattern (OIDRedirectHTTPHandler starts loopback server on random port; OIDExternalUserAgent presents browser). Can be vendored if you don't want to hand-roll the listener; otherwise use it as the design blueprint.
- `Security framework / Keychain` (macOS 26) — Store the per-user refresh token + client_id/secret in the Keychain (separate from the GRDB/SQLCipher data key), so tokens survive restarts and never sit in the encrypted DB or plaintext.

**Patterns / concrete notes**
- Authorization endpoint: https://accounts.google.com/o/oauth2/v2/auth ; Token endpoint: https://oauth2.googleapis.com/token
- Auth request params (verified): client_id, redirect_uri=http://127.0.0.1:<port> (or http://localhost:<port>), response_type=code, scope (space-delimited), code_challenge, code_challenge_method=S256, access_type=offline (required to receive a refresh_token), prompt=consent (forces refresh_token issuance on re-auth)
- Token exchange params (verified): client_id, client_secret (Optional for installed apps), code, code_verifier, grant_type=authorization_code, redirect_uri (must exactly match the loopback URI used in the auth request)
- Loopback flow sequence: (1) open ephemeral NWListener on 127.0.0.1; (2) build redirect_uri from the bound port; (3) ASWebAuthenticationSession opens the auth URL; (4) Google redirects browser to http://127.0.0.1:<port>/?code=...&state=...; (5) the local listener serves a small 'you can close this' HTML page and captures code; (6) exchange code+code_verifier at token endpoint; (7) shut down listener.
- Prefer 127.0.0.1 over localhost for the redirect host — Google docs note localhost may hit client-firewall issues; 127.0.0.1 (and [::1]) is the safer literal.
- Gmail minimization scope: https://www.googleapis.com/auth/gmail.metadata -> users.messages.list + users.messages.get with format=metadata & metadataHeaders=From,Subject,Date returns headers/labels/internalDate but NO body/snippet. (Note: with gmail.metadata you cannot pass q= full-text search; use labelIds and date-bounded list instead.)
- Gmail full-content scope (if brief needs body/snippets): https://www.googleapis.com/auth/gmail.readonly (restricted).
- Calendar minimum read scope: https://www.googleapis.com/auth/calendar.events.readonly (events only) or https://www.googleapis.com/auth/calendar.readonly (calendars + events). Use events.list with timeMin/timeMax bounding to 'today' + singleEvents=true&orderBy=startTime.
- Multi-account Gmail: each Google account = a separate OAuth authorization against the SAME user-owned client_id; store one refresh token per account; add login_hint=<email> on the auth URL to steer the account chooser.
- BYO client setup steps to embed in onboarding UI: (1) console.cloud.google.com -> Create Project; (2) APIs & Services -> Enable APIs: enable 'Google Calendar API' and 'Gmail API'; (3) APIs & Services -> OAuth consent screen -> User type External (personal Gmail) -> fill app name/support email/developer email; (4) add the required scopes; (5) Audience/Test users -> add own email as a Test User (while Testing) OR set publishing status to 'In production'; (6) Credentials -> Create Credentials -> OAuth client ID -> Application type = 'Desktop app'; (7) copy client_id (+ client_secret) into Daybrief.
- 7-day Testing expiry mitigation: detect HTTP 400 invalid_grant with error_description 'Token has been expired or revoked' on refresh; surface a one-tap re-auth; strongly recommend in onboarding that the user flips their own consent screen to 'In production' to make tokens long-lived.

**Gotchas**
- gmail.metadata is RESTRICTED, not non-restricted — it does NOT avoid restricted-scope verification/audience friction. It only reduces the data footprint (a privacy win, and a slightly easier justification narrative), not the verification class.
- ASWebAuthenticationSession CANNOT receive an http://127.0.0.1 loopback redirect — its callbackURLScheme parameter matches a custom (non-http) scheme only. Trying to use it as the redirect catcher for the Google Desktop loopback flow will silently never fire the completion handler. You MUST run a local loopback HTTP listener.
- Testing-status consent screen revokes refresh tokens after 7 days (External user type). For a DAILY app this means it breaks every week unless the user's own app is set to 'In production' — make this an explicit, prominent onboarding step, not a footnote.
- access_type=offline is required to get a refresh_token; Google only returns the refresh_token on the FIRST authorization unless prompt=consent is sent — without prompt=consent a re-auth may return only an access_token and you'll lose offline access.
- redirect_uri must byte-for-byte match between auth request and token exchange, including the ephemeral port — capture the bound port from the listener and reuse the exact string.
- With gmail.metadata you cannot use full-text q= search on messages.list (Google rejects search queries on metadata-only scope); restrict to labelIds + date range instead.
- Desktop OAuth 'client secret' is not confidential in an installed app — do not treat it as a security boundary; PKCE is the real protection. Storing it per-user in the BYO model is acceptable since it's the user's own secret in their own Keychain.
- The loopback flow is vulnerable to a same-machine MITM (another local app racing the loopback interface) — PKCE + a random state value + binding to 127.0.0.1 (not 0.0.0.0) mitigates this; never bind the listener to a public interface.
- If Daybrief itself never sends restricted Gmail data to any backend server (it's local-first), the heavyweight CASA security assessment is NOT triggered for a user who self-verifies — but each user's app still needs to be 'In production' or keep re-consenting every 7 days. CASA is a server-data concern; the 7-day expiry is the actual blocker for the local model.

**Risks**

## Slack BYO-app integration (user-created single-workspace internal app; read mentions + DMs over a 24h window)

**Summary.** The Slack connector IS viable as designed, but only if the user keeps their app INTERNAL — created and installed to a single workspace with public distribution NOT activated. The punishing May 2025 limit (1 request/min, 15 messages max on conversations.history/replies) applies only to apps that have activated public/OAuth distribution outside the Marketplace ("unlisted distributed" apps); Slack explicitly exempts "internal customer-built applications," which retain Tier 3 (50+ req/min, up to 1,000 messages per request). Daybrief must use a USER token (xoxp-), not a bot token: search.messages (the only way to fetch mentions across channels) requires search:read which is a user-token-only scope, and bot tokens cannot search at all. Critically, BOTH search.messages and Slack's newer Real-time Search API explicitly bar "unlisted distributed apps," so the internal/un-distributed posture is mandatory for the connector to function, not just for favorable rate limits. Recommend: user-token (xoxp) auth via the no-OAuth-flow "Install to Workspace" button, search.messages for mentions, conversations.list+conversations.history for DMs, with a polling fallback that degrades gracefully if a user mis-configures their app as distributed.


**Decisions**
- **Use a USER token (xoxp-), not a bot token (xoxb-)** — Mentions must come from search.messages, which requires the search:read scope — a user-token-only legacy scope; bot tokens cannot call search.messages at all. The same user token also reads the user's own DMs/group-DMs (everything the user can see). One token covers both data sources and matches the local-first, act-as-the-user model.
- **Require the user's app to be INTERNAL (single-workspace, public distribution NOT activated)** — Slack's May 2025 changelog exempts 'internal customer-built applications' from the 1-req/min, 15-message limit; they keep Tier 3 (50+/min, 1,000 msgs/request). Activating public distribution flips the app to 'unlisted distributed' and (a) triggers the harsh limits and (b) is explicitly BLOCKED from both search.messages and the Real-time Search API. So internal is required for function, not just speed.
- **Install via the 'Install to Workspace' button (no OAuth web flow / no callback server)** — For a single workspace, Slack issues the xoxp- user token directly on the app's Install App page once User Token Scopes are configured — no redirect URL, no token exchange, no hosted callback. This preserves Daybrief's 'no 24/7 server in the core product' principle. The user pastes the xoxp- token (and team/app metadata) into Daybrief; store the token in the SQLCipher DB / Keychain.
- **Fetch strategy: search.messages for mentions; conversations.list(types=im,mpim) + conversations.history for DMs** — search.messages with a query like 'after:<yesterday>' plus the user's @handle (or in:@self / has:@me semantics) returns mentions and DM matches with type='im' tagging. For exhaustive DM coverage, enumerate IM/MPIM channels via conversations.list(types=im,mpim) then pull each with conversations.history bounded by oldest/latest unix timestamps for the 24h window. groups:history covers private channels the user is in if mentions there are wanted.
- **Minimum user-token scopes: search:read, im:history, mpim:history, users:read** — search:read = mentions via search.messages (covers public/private channel mentions and DMs the user can see). im:history + mpim:history = DM and group-DM message bodies via conversations.history. users:read = resolve author user IDs to display names for the brief. Add channels:history / groups:history ONLY if you also want to scrape channel context beyond search hits; omit them to honor minimum-scope. conversations.list needs no extra scope beyond the *:history scopes for the types requested.
- **Treat the Real-time Search API (assistant.search.context) as a future option, not the v0 path** — It is Slack's recommended replacement for search.messages and is allowed for internal apps, but it is gated to 'AI-enabled' apps, uses a different scope family (search:read.public/.private/.im/.mpim) and an action_token for bot tokens. search.messages has no announced sunset date, works today with a plain user token, and is simpler. Build the mentions fetch behind an interface so it can swap to assistant.search.context later.

**Stack**
- `Slack Web API (slack.com/api)` (2026-current) — Source of mentions (search.messages) and DMs (conversations.list / conversations.history). No official Swift SDK; call REST endpoints directly with URLSession/async-await from the pure-Swift Connector conforming to the Connector protocol.
- `search.messages method` (legacy (no sunset announced as of June 2026)) — Only way to fetch mentions across all channels/DMs with one call. Tier 2 (20+/min). User token + search:read. Returns im-typed results for DMs.
- `conversations.list + conversations.history methods` (current) — Enumerate (types=im,mpim) and read DM/group-DM history. For internal apps: Tier 3, up to 1,000 messages per request, oldest/latest unix-ts bounding for the 24h window. inclusive=true to include boundary messages.
- `users.info / users.list (users:read)` (current) — Resolve author/user IDs returned by messages into display names for the brief. Cache in the local GRDB DB to avoid per-message lookups.
- `assistant.search.context (Real-time Search API)` (current (2025+)) — Slack's recommended successor to search.messages; available to internal apps; AI-app-gated. Keep as a pluggable alternate implementation behind the mentions-fetch interface for forward-compat.

**Patterns / concrete notes**
- App-creation steps (BYO): user goes to api.slack.com/apps -> Create New App -> From scratch -> pick their workspace. Under 'OAuth & Permissions' add USER TOKEN SCOPES (not bot scopes): search:read, im:history, mpim:history, users:read. Do NOT click 'Activate Public Distribution' (keep it internal). Click 'Install to Workspace', approve, copy the 'User OAuth Token' starting with xoxp-. Paste into Daybrief.
- Token discrimination: xoxp- = user token (what Daybrief needs). xoxb- = bot token (cannot search, wrong choice). Validate the pasted token starts with 'xoxp-' and reject xoxb- with a clear error in the connect UI.
- Mentions fetch (24h): POST https://slack.com/api/search.messages with Authorization: Bearer xoxp-..., query='after:2026-06-16' (yesterday's date) optionally AND the user's own @handle; paginate via count (max 100) + page (max 100). Results carry type='im' for DMs. Note: results are affected by the user's Slack-UI search filters, and near-duplicate adjacent matches collapse to one.
- DM enumeration + read: GET conversations.list?types=im,mpim (cursor-paginate) to get DM channel IDs, then for each GET conversations.history?channel=<id>&oldest=<now-86400>&latest=<now>&inclusive=true&limit=200. Internal app => limit up to 1000 honored; distributed app => silently capped at 15 (detection signal — see gotchas).
- Internal-app rate budget: Tier 3 = 50+ req/min, 1000 msgs/request. A 24h sweep of, e.g., 30 DM channels = ~31 calls total — trivially within budget on a manual/periodic refresh. No 24/7 polling needed; fetch on menu-bar open / scheduled brief assembly.
- 429 handling: on HTTP 429 read the Retry-After header (seconds) and back off exactly that long before retrying. Implement as a shared rate-limiter in the Slack Connector so search + history calls share one budget.
- Detect mis-configured (distributed) app at runtime: if conversations.history returns at most 15 messages despite limit=200/1000, OR search.messages / conversations.history returns an access/method error tied to distribution, surface 'Your Slack app appears to be publicly distributed — Daybrief needs an internal (single-workspace) app' guidance.
- Forward-compat alternate: assistant.search.context with scopes search:read.public + (search:read.im, search:read.mpim, search:read.private as needed); user token needs no action_token, bot token does. Swap behind the same MentionsSource interface.

**Gotchas**

## Google Calendar + Gmail fetch & normalization (native Swift)

**Summary.** For a today+tomorrow brief, call Calendar `events.list` once per selected calendar with `singleEvents=true&orderBy=startTime` and an RFC3339 `timeMin`/`timeMax` window; map each Event directly to a BriefItem (htmlLink as the deep link). For Gmail, do a single `messages.list?q=(is:unread OR is:important) newer_than:1d&maxResults=50` (newer_than IS valid Gmail search syntax), then `messages.get?format=metadata&metadataHeaders=From&metadataHeaders=Subject&metadataHeaders=Date` per id (avoid format=full to keep payloads tiny — snippet is already returned even in metadata mode). The dominant cost driver is the N+1 messages.get fan-out: each list+get pair burns 10 Gmail quota units, and Gmail added a NEW per-user 250 units/sec rate limit on 2026-05-01, so bound concurrency to ~5-8 in-flight gets and apply truncated exponential backoff on 429/403 userRateLimitExceeded. Keep payloads light by requesting only the three headers and never the body for v0.


**Decisions**
- **Calendar: one events.list call per calendar (primary + selected), with singleEvents=true & orderBy=startTime over an RFC3339 today-00:00 .. tomorrow-23:59:59 window in the user's local timezone** — orderBy=startTime is ONLY valid when singleEvents=true (expands recurrences into concrete instances). events.list does not accept a list of calendars, so one call per calendarId is required; results are already chronologically sorted, so no client-side sort needed. Window built in local tz so 'today/tomorrow' matches the user's clock.
- **Gmail: messages.list with q="(is:unread OR is:important) newer_than:1d" and maxResults=50** — newer_than:Nd is officially-supported Gmail advanced-search syntax (d/m/y units), verified against Google support docs — cheaper and clearer than computing after:<unixts>. A 24h unread/important window almost always fits in one page for a single user, avoiding pagination entirely.
- **Gmail: messages.get with format=metadata + metadataHeaders=From,Subject,Date (NOT format=full)** — metadata returns id, threadId, labelIds, snippet, internalDate, sizeEstimate and ONLY the requested headers — everything BriefItem needs (title=Subject, people=From, body=snippet, timestamp=internalDate) at a fraction of full's payload. format=full would pull the entire MIME tree/body we don't render in v0. metadata still costs 5 units same as full, so the win is bandwidth/parse-time, not quota.
- **Deep links: Calendar uses event.htmlLink verbatim; Gmail constructs https://mail.google.com/mail/u/0/#all/{message.id}** — htmlLink is an official read-only field that opens the event in the web UI. Gmail has no official permalink field, but the #all/{id} web-UI hash route is the de-facto pattern (id is the hex message id from the API). u/0 assumes the first signed-in account in the browser.
- **Concurrency cap ~5-8 parallel messages.get via a Swift actor / TaskGroup with a semaphore, truncated exponential backoff with jitter on 429 + 403 userRateLimitExceeded/rateLimitExceeded** — The list→per-message-get fan-out is N+1; the NEW 2026-05-01 Gmail 250 units/sec per-user cap means an unbounded TaskGroup of gets (5 units each) can trip the limit at ~50 concurrent. A small bounded pool keeps well under budget and is trivially expressed with Swift 6 structured concurrency.
- **Use Calendar fields= and Gmail format=metadata to request the minimum fields only** — Calendar partial response (fields=items(id,status,htmlLink,summary,location,start,end,attendees(email,displayName,responseStatus,self),organizer,hangoutLink,conferenceData,recurringEventId),nextPageToken) drops large unused fields and shrinks the JSON the Swift Codable layer must decode.

**Stack**
- `Google Calendar API v3 events.list` (v3 (rev ~20260225)) — Primary source for calendar events; supports singleEvents/orderBy/timeMin/timeMax and returns htmlLink for deep links.
- `Gmail API v1 users.messages.list / users.messages.get` (v1 (rev ~20260427)) — List by search query then fetch headers+snippet; format=metadata + metadataHeaders is the cheap path.
- `URLSession + async/await (Foundation)` (macOS 26 SDK) — No third-party Google SDK needed for plain REST GETs; URLSession.data(for:) with a bearer token is sufficient and keeps the binary lean. Decode with Codable structs mirroring Event and Message.

**Patterns / concrete notes**
- Calendar request (per calendarId): GET https://www.googleapis.com/calendar/v3/calendars/{calendarId}/events?singleEvents=true&orderBy=startTime&timeMin=2026-06-17T00:00:00-07:00&timeMax=2026-06-18T23:59:59-07:00&maxResults=250&fields=items(id,status,htmlLink,summary,location,start,end,attendees(email,displayName,responseStatus,self),organizer,hangoutLink,conferenceData,recurringEventId),nextPageToken . Header: Authorization: Bearer <token>. timeMin/timeMax MUST be RFC3339 WITH offset (e.g. -07:00 or Z); milliseconds ignored; timeMax must be > timeMin.
- Calendar response: { items: [Event], nextPageToken? }. Each Event: { id, status('confirmed'|'tentative'|'cancelled'), htmlLink, summary, description?, location?, start:{dateTime?,date?,timeZone?}, end:{...}, attendees?:[{email,displayName?,responseStatus,self?,organizer?}], organizer:{email,displayName?,self?}, hangoutLink?, conferenceData?, recurringEventId? }. All-day events use start.date 'yyyy-mm-dd' (no time); timed events use start.dateTime RFC3339. SKIP status=='cancelled' instances.
- Gmail list request: GET https://gmail.googleapis.com/gmail/v1/users/me/messages?q=(is:unread%20OR%20is:important)%20newer_than:1d&maxResults=50 . Response: { messages:[{id,threadId}], nextPageToken?, resultSizeEstimate }. messages[] contains ONLY id+threadId — a second get is mandatory for headers.
- Gmail get request: GET https://gmail.googleapis.com/gmail/v1/users/me/messages/{id}?format=metadata&metadataHeaders=From&metadataHeaders=Subject&metadataHeaders=Date . Response Message: { id, threadId, labelIds:[String], snippet:String, payload:{ headers:[{name,value}] }, internalDate:'<epoch-ms-as-string-int64>', sizeEstimate }. Extract From/Subject/Date by case-insensitive match on header.name within payload.headers.
- internalDate is an int64 epoch-MILLISECONDS string — parse as Int64 then Date(timeIntervalSince1970: ms/1000). Prefer internalDate over the Date header for sorting (Date header is sender-supplied and unreliable).
- From header value is RFC5322 form 'Display Name <addr@host>' or bare 'addr@host'. Parse into people:[{name?,email}] — split on the last '<','>' pair; fall back to the raw address as both name and email.
- BriefItem normalization — Calendar Event -> { source:'google_calendar', account:<oauth account email>, space:<calendar summary or calendarId>, type:'event', title: summary ?? '(No title)', body: location ?? description-first-line?, people: attendees.map{name/email}, timestamp: start.dateTime ?? startOfDay(start.date), url: htmlLink, urgencyHints: derive (e.g. ['starting_soon'] if start within 30m, ['all_day'] if start.date present, ['needs_response'] if self.responseStatus=='needsAction', ['has_video'] if hangoutLink|conferenceData) }.
- BriefItem normalization — Gmail Message -> { source:'gmail', account:<oauth account email>, space:'inbox' (or first user label), type:'email', title: Subject ?? '(no subject)', body: snippet, people: [parsed From], timestamp: Date(internalDate ms), url: 'https://mail.google.com/mail/u/0/#all/'+id, urgencyHints: from labelIds — ['unread'] if 'UNREAD' in labelIds, ['important'] if 'IMPORTANT' in labelIds, ['starred'] if 'STARRED' }.
- Pagination loop (both APIs): while let token = resp.nextPageToken { refetch with &pageToken=token }. For Gmail also pass the SAME q on each page. In practice a 24h single-user window rarely paginates; cap total fetched at e.g. 100 messages to keep the brief bounded and quota low.
- Quota-aware fan-out (Swift 6): bound messages.get with a counting actor/semaphore (maxInFlight 5-8) inside a withThrowingTaskGroup; on HTTP 429 or 403 with reason userRateLimitExceeded/rateLimitExceeded, sleep min((2^n)+jitterMs, 32s) then retry the single failed get, not the whole batch.
- Timezone: compute timeMin/timeMax using Calendar.current + TimeZone.current; emit RFC3339 with ISO8601DateFormatter([.withInternetDateTime]) (which includes the offset). Do NOT use 'Z' unless converting to UTC, or 'today' will be wrong near midnight.

**Gotchas**
- orderBy=startTime is REJECTED unless singleEvents=true — and orderBy/timeMin/timeMax CANNOT be combined with nextSyncToken. For a fresh today+tomorrow pull you don't use sync tokens, so this is fine, but a future incremental-sync mode must drop orderBy.
- timeMin/timeMax MUST include a timezone offset (RFC3339) or the request errors / behaves unexpectedly; bare 'yyyy-MM-ddTHH:mm:ss' is invalid. Milliseconds are accepted but ignored.
- Recurring events: with singleEvents=true you get concrete instances (each with its own start, and a recurringEventId pointing at the master). Without it you'd get the master with a recurrence rule and no usable per-day start — always set singleEvents=true for a brief.
- Gmail messages.list returns ONLY {id, threadId} per message — there is NO way to get From/Subject/snippet in the list call; the per-message get is unavoidable (this is the N+1 cost).
- format=metadata and format=full BOTH cost 5 quota units — choosing metadata saves bandwidth/parse time, NOT quota. The real quota lever is reducing the NUMBER of messages fetched (tight q + maxResults cap).
- NEW 2026-05-01 Gmail per-user rate limit of 250 quota units/sec (previously effectively only a daily cap). messages.list=5 and messages.get=5 units each, so ~25 messages fully fetched per second is the safe ceiling per user — bound concurrency accordingly.
- internalDate is epoch MILLISECONDS as a STRING (int64), not seconds and not RFC3339 — divide by 1000 before Date(timeIntervalSince1970:). Easy off-by-1000x bug.
- The Gmail #all/{id} deep link is NOT an officially documented/stable URL contract and assumes account index u/0 (the first signed-in browser account). If the user's brief account isn't u/0, the link opens the wrong mailbox or 404s. Consider deriving the index or using https://mail.google.com/mail/?authuser=<email>#all/{id}.
- htmlLink for Calendar opens the event correctly but also implicitly uses the browser's signed-in account; multi-account users may land in the wrong account. Same authuser concern as Gmail.
- is:important depends on Gmail's Priority Inbox being enabled; if the user disabled importance markers, the IMPORTANT label/operator returns little — the brief should degrade gracefully to is:unread.
- Multi-account (the spec requires multi-account Gmail): each account has its OWN token, its OWN /users/me, AND its OWN 250 units/sec budget — run accounts in parallel but keep each account's get-pool bounded independently; set BriefItem.account from the OAuth identity, not from the API response.
- Cancelled events still appear in events.list (status=='cancelled') for instances that were declined/removed from a recurring series — filter them out or they pollute the brief.

**Risks**
- [high] newer_than:1d is valid Gmail API q syntax (d/m/y units) — Confirmed against Google's official Gmail advanced-search support doc (support.google.com/mail/answer/7190): 'Search for emails older or newer than a time period. Use d (day), m (month), or y (year).' The Gmail API q parameter uses the same syntax as the search box.
- [high] messages.list costs 5 units, messages.get costs 5 units; Gmail enforces 250 units/sec per user as of 2026-05-01 — Per Gmail API quota docs and 2026 reference guides: list=5, get=5, send=100; the 250 units/user/second per-user rate limit was introduced 2026-05-01 (previously daily-only).
- [medium] Calendar default per-user-per-project quota is 600 requests/min and per-project is 10,000/min — Calendar quota guide (developers.google.com/workspace/calendar/api/guides/quota) cites 10,000 req/min/project and 600 req/min/user/project, but exact defaults vary by project and Google adjusts them; verify in the project's Cloud Console before relying on the number.
- [medium] The https://mail.google.com/mail/u/0/#all/{id} deep link reliably opens the specific message — It is the widely-used de-facto pattern (labnol, Gmail community threads) and the API hex message id works in the #all/ route, but Google does NOT document this as a stable contract and it assumes browser account index u/0. Treat as best-effort; add authuser=<email> to harden.
- [high] format=metadata returns the snippet field — Gmail Message resource: snippet is a top-level Message field returned for metadata and full formats (only minimal/raw differ); metadataHeaders only restricts payload.headers, not snippet.
- [high] events.list cannot query multiple calendars in one call — events.list is scoped to a single {calendarId} path parameter; the API has no multi-calendar batch in one events.list request. One request per calendar (or a batch HTTP request) is required.

## LLM adapter (bring-your-own model) in Swift

**Summary.** Define one `ModelAdapter` protocol with `complete()` and `streamComplete()` (returning `AsyncThrowingStream<String, Error>`) plus a `completeStructured()` variant, and a `ProviderRegistry` that maps a provider id + user config to a concrete adapter. Implement five backends as pure-Swift `Sendable` actors/structs: OpenRouter (default), OpenAI, Anthropic, Google Gemini, and local Ollama. All five share an SSE reader built on `URLSession.bytes(for:)` iterating `.lines` and parsing `data:` frames (Ollama uses NDJSON, not SSE — handle as a sibling). For structured brief output, standardize OpenRouter on `response_format: {type:"json_schema", json_schema:{name, strict:true, schema}}` with `provider.require_parameters:true`, and ALWAYS run the parsed JSON through a provider-agnostic validate-and-repair layer as the safety net, because OpenRouter passthrough fidelity varies by underlying model. The synthesis system prompt and the render template should live as user-editable files on disk (e.g. `~/Library/Application Support/Daybrief/prompts/`), loaded at runtime with bundled defaults as fallback.


**Decisions**
- **Single protocol `ModelAdapter` with `complete(system:messages:) async throws -> String`, `streamComplete(...) -> AsyncThrowingStream<String, Error>`, and `completeStructured<T: Decodable & Sendable>(system:messages:schema:) async throws -> T`** — One protocol covers all three call shapes the app needs (one-shot brief synthesis, v1 streaming chat, structured brief JSON). Keeping structured output a first-class protocol method lets each adapter use its provider's NATIVE schema enforcement instead of forcing every backend through string parsing.
- **Use `AsyncThrowingStream<String, Error>` (token deltas) for streaming, not AsyncStream** — Streaming HTTP can fail mid-flight (network drop, provider 5xx, malformed SSE). AsyncStream cannot surface errors, so a non-throwing stream would have to swallow failures or smuggle them in-band. AsyncThrowingStream lets the UI distinguish 'stream finished' from 'stream broke'.
- **Standardize the DEFAULT (OpenRouter) on `response_format` json_schema with `strict:true` + `provider.require_parameters:true`, backed by a universal parse-and-repair fallback** — OpenRouter normalizes OpenAI-style `response_format` across providers, but enforcement quality depends on the routed underlying model — some open models only soft-honor the schema. `require_parameters:true` filters routing to models that actually support it; the repair layer catches the residual failures. This is the most portable strategy that still gets hard guarantees on capable models.
- **Each provider implemented as a separate type conforming to ModelAdapter, resolved via a `ProviderRegistry` keyed by a `ProviderID` enum** — Adapters differ in endpoint, auth header, request body shape, structured-output mechanism, and stream framing (SSE vs NDJSON). A registry keeps the differences isolated per-adapter and lets the app instantiate the right one from saved user config (provider id + API key from Keychain + model id + optional base URL). New providers = add one type + one registry case.
- **Make adapters `Sendable` value types or `actor`s; mark protocol methods `async throws` explicitly; avoid global-actor isolation on the protocol** — Swift 6.2 strict concurrency rejects actor-isolated types satisfying isolation-agnostic protocol requirements. Declaring the protocol methods plainly `async throws` (caller-isolation via default) and making each adapter independently Sendable lets the SynthesisService call adapters from any context without data-race diagnostics. API-key strings and configs crossing into the adapter must be Sendable.
- **Prompt + render template as on-disk user-editable files with bundled defaults** — Local-first product principle: the user owns and can tune the synthesis behavior without recompiling. Load `synthesis.system.md` and `brief.template` from Application Support at runtime; ship read-only defaults in the app bundle and copy-on-first-launch so the user always has an editable copy and a known-good fallback.

**Stack**
- `URLSession (Foundation)` (macOS 26 SDK) — Native HTTP + `bytes(for:)` async byte stream for SSE/NDJSON. No third-party networking dependency needed; keeps the dependency surface minimal for an open-source local-first app.
- `Swift Concurrency (async/await, AsyncThrowingStream)` (Swift 6.2) — Protocol async methods and the streaming variant. Swift 6.2 'approachable concurrency' (nonisolated(nonsending), Sendable inference) reduces friction for passing configs/keys into adapters under strict concurrency.
- `Codable / JSONEncoder / JSONDecoder (Foundation)` (macOS 26 SDK) — Encode provider-specific request bodies and decode the brief JSON. The repair layer also re-decodes after fixups.
- `mattt/EventSource (optional)` (latest 2026) — Spec-compliant SSE client with AsyncSequence support if hand-rolling the SSE frame parser proves brittle (handles id/event/data/retry and LF/CR/CRLF). Optional — a ~40-line custom `.lines` parser usually suffices since all five providers emit simple `data:`-only frames.

**Patterns / concrete notes**
- OpenRouter chat: POST https://openrouter.ai/api/v1/chat/completions, headers `Authorization: Bearer <key>`, `Content-Type: application/json`, optional `HTTP-Referer: <app url>` and `X-Title: Daybrief` (HTTP-Referer is REQUIRED for ranking/app-page; X-Title alone does nothing). Body: {model, messages:[{role,content}], stream, response_format}. Models list: GET https://openrouter.ai/api/v1/models (and filter `supported_parameters=structured_outputs` to gate structured mode).
- OpenRouter structured output (DEFAULT strategy): response_format = {"type":"json_schema","json_schema":{"name":"daily_brief","strict":true,"schema":{...}}}. Add provider preferences {"require_parameters":true} so routing only picks models that honor it. Set `additionalProperties:false` and list ALL properties in `required` on every object (OpenAI strict-mode rule that OpenRouter inherits). Optional Response Healing plugin reduces invalid-JSON risk on non-streaming requests.
- Anthropic Messages: POST https://api.anthropic.com/v1/messages, headers `x-api-key: <key>`, `anthropic-version: 2023-06-01`, `Content-Type: application/json`. Body: {model, max_tokens, system, messages, stream}. NATIVE structured output is now GA via `output_config: {format: {type:"json_schema", schema:{...}}}` (constrained decoding, guarantees schema compliance; the older `output_format` + `structured-outputs-2025-11-13` beta header still works but is deprecated/unnecessary). Refusals (stop_reason "refusal") may not match schema. Fallback for older models: forced tool-use — define a single tool with `strict:true` + input_schema, set tool_choice {type:"tool", name:"emit_brief"}, read the tool_use block's input.
- OpenAI: POST https://api.openai.com/v1/chat/completions (Chat Completions still supported; Responses API is the newer migration target), `Authorization: Bearer <key>`. Structured output: response_format {type:"json_schema", json_schema:{name, strict:true, schema}} — REQUIRES `additionalProperties:false` and every field in `required` on all nested objects, else strict mode errors.
- Google Gemini: POST https://generativelanguage.googleapis.com/v1beta/models/<model>:generateContent?key=<key> (use :streamGenerateContent for streaming). Body uses {contents:[{role,parts:[{text}]}], systemInstruction, generationConfig}. Structured output: generationConfig {responseMimeType:"application/json", responseSchema:{...}}. Note role/parts shape differs from OpenAI; map messages accordingly.
- Ollama (local): POST http://localhost:11434/api/chat, no auth. Body {model, messages:[{role,content}], stream, format}. Structured output: `format` is the raw JSON schema object directly (NOT wrapped in response_format/json_schema). Streaming = NDJSON (newline-delimited JSON objects), each line a partial; final object has "done":true — this is NOT SSE, so the stream reader must branch on provider framing.
- SSE reader pattern: `let (bytes, response) = try await session.bytes(for: request); for try await line in bytes.lines { guard line.hasPrefix("data: ") else { continue }; let payload = line.dropFirst(6); if payload == "[DONE]" { break }; decode JSON chunk, extract choices[0].delta.content, yield into the AsyncThrowingStream continuation }`. `.lines` buffers to newline so partial frames don't split. Check the HTTPURLResponse status before iterating; on non-2xx, drain and throw a typed error.
- Provider-agnostic repair fallback: (1) try native structured output; (2) if unavailable/disabled, append a JSON-only instruction + the schema to the system prompt; (3) decode the raw string; (4) on decode failure, strip markdown code fences / leading prose, extract the first balanced {...} or [...] span, and re-decode; (5) on second failure, optionally re-ask the model with the parse error appended (one bounded retry) before surfacing an error.
- ProviderRegistry: `enum ProviderID { case openRouter, openAI, anthropic, gemini, ollama }`; `struct ProviderConfig: Sendable { let id: ProviderID; let apiKey: String?; let model: String; let baseURLOverride: URL? }`; `func makeAdapter(_ cfg: ProviderConfig) -> any ModelAdapter`. App reads ProviderConfig from settings, pulls the key from Keychain, constructs the adapter on demand.
- Editable config files: on first launch copy bundled `synthesis.system.md` and `brief.template` into ~/Library/Application Support/Daybrief/prompts/; load with try-disk-then-bundle fallback; expose 'Reset to default' (re-copy from bundle). Template variables filled from the decoded brief struct before rendering.

**Gotchas**
- OpenAI/OpenRouter strict json_schema mode REQUIRES `additionalProperties:false` on every object AND every property listed in `required` — optional fields must be modeled as nullable (type:["string","null"]) not omitted, or the request 400s. This shapes how you define the DailyBrief schema.
- OpenRouter `response_format` is normalized but enforcement depends on the routed underlying model; without `provider.require_parameters:true` a request can be routed to a model that ignores the schema and returns prose. Always set require_parameters AND keep the repair fallback — this is why repair is mandatory, not optional.
- Ollama does NOT use `response_format`/`json_schema` wrapper — it takes the bare JSON schema in a top-level `format` field, and it streams NDJSON not SSE. Treating Ollama like an OpenAI-compatible endpoint for streaming/structured output will silently break.
- Anthropic uses `x-api-key` + `anthropic-version` headers (NOT `Authorization: Bearer`) and `system` is a top-level field, not a message role. Its SSE event format is richer (event: types like content_block_delta) — the Anthropic stream parser must read `event:` lines and JSON `data:`, not just concatenate deltas like OpenAI.
- Anthropic native structured output (`output_config.format`) is GA and uses constrained decoding (real guarantee), but is only on recent Claude models; older models need the forced-tool-use fallback. Don't hardcode one path — pick per-model capability.
- Gemini's message shape is {contents:[{role:'user'|'model', parts:[{text}]}]} with a separate `systemInstruction` — the role names and nesting differ from OpenAI's {role:'system'|'user'|'assistant', content}. The adapter must translate the canonical message list, not pass it through.
- Swift 6.2 strict concurrency: a `@MainActor`-isolated or actor-isolated adapter cannot conform to an isolation-agnostic protocol requirement without a diagnostic. Keep the protocol methods plainly `async throws` and make adapters Sendable value types (or use `nonisolated`) so the SynthesisService can call them off the main actor. API keys/configs crossing the boundary must be Sendable.
- SSE `.lines` handles most framing, but providers differ on keep-alive comments (lines starting with `:`), `[DONE]` sentinels (OpenAI/OpenRouter send it; Anthropic/Gemini/Ollama do not), and whether the final usage chunk arrives after content. Per-provider stream termination logic is required.
- OpenRouter HTTP-Referer is the load-bearing attribution header; X-Title alone creates no app page. For a local desktop app there is no real referer URL — use a stable identifier (e.g. the GitHub repo URL) or omit attribution entirely since it's optional for functionality.
- Token deltas can split a multi-byte UTF-8 character or a JSON structural boundary across chunks. For the streaming CHAT use case yielding raw string deltas is fine; but do NOT attempt to incrementally JSON-parse a streamed structured brief — accumulate the full body then decode once.

## App architecture, module boundaries, and the connector orchestrator (SPM workspace layout, Connector protocol, structured-concurrency pipeline, @Observable state, DI, fixture testing)

**Summary.** Use an Xcode app target ("Daybrief") that owns only the menu-bar shell (MenuBarExtra/NSStatusItem) and SwiftUI composition root, with all logic in local SPM packages wired by a single root Package. Recommended packages: DaybriefCore (domain models + errors, pure value types, Sendable), Persistence (GRDB 7 over SQLCipher, exposes DatabaseReader/DatabaseWriter as the injection seam), Secrets (Keychain + token store), ConnectorKit (the Connector protocol + OAuth/Account abstractions + fixture test harness), per-connector packages (GoogleCalendarConnector, GmailConnector, SlackConnector) each depending only on ConnectorKit + DaybriefCore, LLMKit (BYO-model client), Pipeline (the orchestrator), BriefRender (markdown/section assembly), and AppFeature (the @Observable AppModel + SwiftUI views). The orchestrator runs every connector's fetch concurrently in a withThrowingTaskGroup where each child task internally races the fetch against a per-connector timeout and converts all outcomes to a non-throwing ConnectorOutcome (success/timeout/failure) so a dead or slow connector can never throw out of the group or kill the brief — the pipeline always assembles a partial brief plus a surfaced list of per-connector errors. Critical version facts: Swift 6.2/Xcode 26 makes the APP target MainActor-isolated by default while SPM packages stay nonisolated unless they opt in via .defaultIsolation(MainActor.self) per target — so logic packages stay off the main actor and only AppFeature opts into MainActor.


**Decisions**
- **Xcode app target + ~9 local SPM packages declared in one root Package.swift (or an .xcworkspace referencing a Packages/ dir); app target depends only on AppFeature, which is the composition root.** — Keeps the app target thin (menu-bar shell, entitlements, Info.plist only) and forces all logic into testable, dependency-bounded packages. SPM enforces the dependency graph at compile time, so a connector physically cannot import the UI layer. A single root package is simpler to maintain than nested package-per-folder for a solo/small team while preserving module boundaries.
- **Connector protocol lives in ConnectorKit and is abstract enough to be backed by an out-of-process runner later: async fetch(account:since:) -> [NormalizedItem], plus value-type OAuthConfig, Account, and a ConnectorDescriptor. All inputs/outputs are Sendable Codable value types; no live network/SDK types cross the boundary.** — Because the protocol only exchanges Sendable Codable values (not SDK objects or live sockets), the exact same protocol can later be satisfied by an XPC/subprocess proxy that serializes calls to an out-of-process runner — no call-site changes. This directly satisfies the locked 'keep abstract enough to back with an out-of-process runner' requirement and the future JS-connector escape hatch without committing to it now.
- **Orchestrator uses withThrowingTaskGroup, but each child task is itself non-throwing: it races connector.fetch against a Task.sleep(for: timeout) inside an inner withThrowingTaskGroup (first-wins, then cancel the loser) and maps any throw/timeout to a ConnectorOutcome enum. The OUTER group only ever receives ConnectorOutcome values.** — In Swift structured concurrency, if any child task throws out of withThrowingTaskGroup, the group cancels ALL siblings and rethrows — which would let one dead connector kill the entire brief. By catching at the child boundary and returning a Result-like ConnectorOutcome, every connector runs to completion or timeout independently, guaranteeing partial-brief assembly with surfaced errors. Verified against SE-0304 and current TaskGroup guidance.
- **Persistence exposes GRDB's DatabaseReader/DatabaseWriter protocols (both Sendable in GRDB 7) as the public injection type — NOT concrete DatabaseQueue/DatabasePool. Production wires a SQLCipher DatabasePool; tests/previews wire an in-memory DatabaseQueue.** — DatabaseReader and DatabaseWriter are the canonical GRDB DI seam (both are AnyObject+Sendable protocols adopted by DatabaseQueue and DatabasePool). Depending on the protocol lets tests inject an in-memory DatabaseQueue with the exact same migrations, giving deterministic, fast, fully-isolated DB tests with zero filesystem/Keychain dependency.
- **App state is a single @Observable AppModel (in AppFeature, MainActor-isolated) holding @ObservationIgnored references to injected services (Pipeline, Persistence reader, Secrets) and publishing observable view state (brief, per-connector status, generationPhase). Long work runs on nonisolated package code via await; results hop back to MainActor for the UI.** — @Observable + SwiftUI is the 2026-current state pattern (replaces ObservableObject). Keeping AppModel on MainActor matches Xcode 26's default app-target isolation, while the nonisolated logic packages keep heavy I/O off the main thread. @ObservationIgnored on injected services prevents spurious view invalidation. This is the smallest-surface, most idiomatic feed into SwiftUI.
- **Dependency injection via constructor injection of protocol abstractions, optionally formalized with Point-Free swift-dependencies (@Dependency/DependencyValues) for ambient services like Clock, network session, and Keychain.** — Plain protocol + initializer injection is sufficient and dependency-free for the connector/persistence seams. swift-dependencies adds value specifically for cross-cutting ambient deps (clock, URLSession, secrets) with scoped test overrides (withDependencies) and is the de-facto 2026 community standard; recommended as an OPTIONAL layer to avoid threading a clock/session through every initializer. Either is acceptable — lead with protocols, adopt the library only if injection boilerplate grows.
- **Inject a Swift Clock (ContinuousClock in prod, Point-Free TestClock in tests) into both the per-connector timeout and the retry/backoff policy.** — Hardcoded Task.sleep makes timeout and exponential-backoff logic untestable (tests would wait real seconds). Injecting a Clock lets TestClock advance virtual time, so backoff/timeout behavior is verified deterministically in milliseconds. This is the current canonical approach (swift-clocks) for testable time-based concurrency.
- **Connectors are tested against recorded JSON fixtures via an injected transport abstraction (e.g. a HTTPTransport protocol or URLProtocol stub) plus a FixtureLoader in ConnectorKit's test support; fixtures are committed under each connector package's Tests/Fixtures.** — Recording real Google/Slack JSON once and replaying it makes connector normalize() logic testable offline, deterministically, with no OAuth or network. Putting the FixtureLoader/transport-stub in ConnectorKit (or a ConnectorTestSupport target) lets every per-connector package reuse the same harness, satisfying the 'test connectors against recorded JSON fixtures' requirement.
- **OAuth handled in Secrets/ConnectorKit using ASWebAuthenticationSession (AuthenticationServices) + PKCE; tokens stored in Keychain via Secrets. SQLCipher key also in Keychain.** — ASWebAuthenticationSession is Apple's recommended native OAuth flow and is what AppAuth itself wraps; combined with PKCE it satisfies the minimum-scope, no-embedded-webview, private-by-default principles for Google + Slack. Keeping token + DB-key storage in a dedicated Secrets package isolates all Keychain access behind one mockable protocol.

**Stack**
- `Swift` (6.2) — Strict concurrency baseline; SE-0466 default actor isolation, @concurrent, and caller-inherited nonisolated async are the foundation of the actor-isolation strategy. Set swift-tools-version:6.2 so .defaultIsolation is available.
- `Xcode` (26) — New app targets default SWIFT_DEFAULT_ACTOR_ISOLATION to MainActor; required for macOS 26 'Tahoe' SDK and Swift 6.2 toolchain.
- `GRDB.swift` (7.10.0 (Feb 2026)) — SQLite access layer; Swift 6.1+ ready with Sendable DatabaseReader/DatabaseWriter protocols (the persistence DI seam), async read/write, and ValueObservation. Build with the SQLCipher trait for the encrypted DB file.
- `AuthenticationServices (ASWebAuthenticationSession)` (macOS 26 SDK) — Native OAuth (Google Calendar/Gmail, Slack) with PKCE and no embedded webview; system-managed auth session keeps the flow private and minimal-scope.
- `swift-dependencies (Point-Free)` (latest (1.x, 2026)) — OPTIONAL ambient-dependency injection (@Dependency/DependencyValues) with scoped test overrides; recommended for Clock/URLSession/Keychain seams if constructor injection gets heavy.
- `swift-clocks (Point-Free) — TestClock` (latest (2026)) — Deterministic virtual-time testing of per-connector timeouts and exponential backoff; ContinuousClock in production, TestClock in tests.
- `Swift Testing` (bundled with Swift 6.2) — Modern @Test/#expect framework for the fixture-replay connector tests and orchestrator partial-brief tests; first-class async support.

**Patterns / concrete notes**
- Root Package.swift declares targets: DaybriefCore, Persistence, Secrets, ConnectorKit, ConnectorTestSupport, GoogleCalendarConnector, GmailConnector, SlackConnector, LLMKit, Pipeline, BriefRender, AppFeature; the Xcode app target ('Daybrief') depends ONLY on AppFeature.
- Dependency graph (arrows = 'depends on'): app -> AppFeature -> {Pipeline, BriefRender, Persistence, Secrets}. Pipeline -> {ConnectorKit, GoogleCalendarConnector, GmailConnector, SlackConnector, LLMKit, DaybriefCore, Persistence}. Each per-connector package -> {ConnectorKit, DaybriefCore, Secrets}. ConnectorKit -> DaybriefCore. Persistence -> {DaybriefCore, GRDB}. Secrets -> DaybriefCore. LLMKit -> DaybriefCore. BriefRender -> DaybriefCore. DaybriefCore -> (no internal deps). Acyclic; DaybriefCore is the shared leaf.
- Per-target isolation in Package.swift: only AppFeature gets swiftSettings: [.defaultIsolation(MainActor.self)]; all logic packages omit it (stay nonisolated by default) so fetch/normalize/DB work runs off the main actor. Verified: a new SPM package does NOT inherit the app target's MainActor default — it must opt in per target.
- Connector protocol (ConnectorKit):\npublic protocol Connector: Sendable {\n  static var descriptor: ConnectorDescriptor { get }\n  var oauthConfig: OAuthConfig { get }\n  func accounts() async throws -> [Account]\n  func fetch(account: Account, since: Date, clock: any Clock<Duration>) async throws -> [NormalizedItem]\n}\nAll of NormalizedItem, Account, OAuthConfig, ConnectorDescriptor are public Sendable Codable structs in DaybriefCore — no SDK/network types cross the boundary (enables future out-of-process runner via XPC serialization).
- OAuthConfig value type: struct OAuthConfig: Sendable { let clientID: String; let authorizationEndpoint: URL; let tokenEndpoint: URL; let redirectURI: URL; let scopes: [String]; let usesPKCE: Bool }. Account: struct Account: Sendable, Codable, Identifiable { let id: AccountID; let connectorID: ConnectorID; let displayName: String; /* token ref is a Keychain key, NOT the token */ let tokenRef: String }. Multi-account = [Account] returned by accounts(); orchestrator fans out over (connector, account) pairs.
- ConnectorOutcome (the partial-brief enabler): enum ConnectorOutcome: Sendable { case success(connectorID: ConnectorID, accountID: AccountID, items: [NormalizedItem]); case timedOut(ConnectorID, AccountID); case failed(ConnectorID, AccountID, ConnectorError) }. Children NEVER throw out of the group — they return this.
- Orchestrator skeleton (Pipeline, nonisolated):\nfunc assembleBrief(...) async -> BriefDraft {\n  await withTaskGroup(of: ConnectorOutcome.self) { group in\n    for job in jobs {  // (connector, account) pairs\n      group.addTask { await self.runWithTimeout(job) }\n    }\n    var outcomes: [ConnectorOutcome] = []\n    for await o in group { outcomes.append(o) }\n    return BriefDraft(items: outcomes.successItems, errors: outcomes.errors)\n  }\n}\nNote: OUTER group is withTaskGroup (non-throwing) precisely so no sibling cancellation cascade can occur.
- Per-connector timeout race (inside runWithTimeout): use an inner withThrowingTaskGroup(of: [NormalizedItem].self) with two children — the real fetch, and a timeout child that does try await clock.sleep(for: timeout) then throws TimeoutError. group.next() takes the first; group.cancelAll() cancels the loser. Wrap in do/catch to map throw -> .timedOut/.failed. Clock injected for TestClock determinism.
- Retry/backoff policy (ConnectorKit): a RetryPolicy struct { maxAttempts: Int; baseDelay: Duration; maxDelay: Duration; jitter: Bool } with func backoff<T>(clock:operation:) that retries only on transient ConnectorError cases (network/5xx/429-with-Retry-After), computing delay = min(maxDelay, baseDelay * 2^attempt) plus optional jitter, sleeping via the injected clock, and respecting Task.isCancelled between attempts. Non-transient errors (401/403/scope) fail fast and surface to the brief.
- @Observable app state (AppFeature, @MainActor):\n@Observable final class AppModel {\n  private(set) var brief: RenderedBrief?\n  private(set) var connectorStatuses: [ConnectorID: ConnectorStatus]\n  private(set) var phase: GenerationPhase  // idle/fetching/synthesizing/done/failed\n  @ObservationIgnored private let pipeline: BriefPipeline\n  @ObservationIgnored private let store: any DatabaseReader\n  func generate() async { phase = .fetching; let draft = await pipeline.assembleBrief(...); ... }\n}\nViews read AppModel directly; service refs are @ObservationIgnored to avoid spurious invalidation.
- Menu-bar shell: App struct uses MenuBarExtra (SwiftUI) with .menuBarExtraStyle(.window) for the brief popover; the App owns the composition root that constructs concrete services (SQLCipher DatabasePool, KeychainSecrets, live connectors) and injects them into AppModel. No 24/7 server — generation is user-triggered or scheduled via a lightweight timer/BGTask, honoring the local-first principle.
- Persistence DI seam: public protocol surface is GRDB's any DatabaseWriter (read+write) / any DatabaseReader; Persistence exposes makeProductionWriter(url:keychainKey:) -> any DatabaseWriter (SQLCipher DatabasePool + migrations) and makeInMemoryWriter() -> any DatabaseWriter (DatabaseQueue + same migrations) for tests. ValueObservation can drive @Observable view state when the brief is persisted.
- Fixture testing harness (ConnectorTestSupport): an HTTPTransport protocol { func data(for: URLRequest) async throws -> (Data, HTTPURLResponse) } injected into each connector; tests provide a FixtureTransport that maps request path -> committed JSON under Tests/Fixtures/<connector>/*.json. Tests assert normalize() output for known fixtures; orchestrator tests inject a connector double that returns .timedOut/.failed to prove partial-brief assembly.

**Gotchas**
- Swift 6.2 behavior change: a nonisolated async function now INHERITS the caller's actor instead of hopping to the global executor. In nonisolated logic packages this means your fetch could end up running on whatever actor called it; mark genuinely-parallel work @concurrent (or push it into the task group) to force the global executor, or you can accidentally serialize all connector work on one actor.
- A new SPM package does NOT inherit the app target's SWIFT_DEFAULT_ACTOR_ISOLATION=MainActor. Packages default to nonisolated. If you forget to add .defaultIsolation(MainActor.self) to AppFeature, your @Observable AppModel and views won't be MainActor by default and you'll get isolation errors; conversely if you accidentally add it to a logic package, all its I/O gets pinned to MainActor and blocks the UI.
- If you use withThrowingTaskGroup for the OUTER orchestration loop and let a child throw, the group cancels ALL siblings and rethrows — exactly the 'one dead connector kills the brief' failure. The fix is the non-throwing outer withTaskGroup + ConnectorOutcome pattern; do not 'simplify' it back to a throwing group.
- Timeout via Task.sleep without a winner-cancel will leak the loser task and waste the connector's network call. After group.next() returns the first result in the inner timeout race, you MUST group.cancelAll() (or rely on the group's implicit cancel-on-return) AND ensure the connector's URLSession call actually checks cancellation, or a hung socket keeps running.
- GRDB record types and associations were not Sendable before GRDB 7; on 7.x DatabaseReader/DatabaseWriter are Sendable but you still must keep mutable Database access inside the read/write closures — never capture a Database handle and use it across an await. Cross only Sendable value snapshots out of the closure.
- Google deprecated the loopback-IP redirect for several OAuth client types; use a custom URI scheme (or the system-handled ASWebAuthenticationSession callback) for the desktop flow, and register the right client type, or the authorization will be rejected.
- Slack validates the User-Agent against supported browsers; an embedded WKWebView UA can be rejected. ASWebAuthenticationSession uses the system browser/SFAuthenticationViewController and avoids this — do not roll your own WKWebView OAuth.
- Hardcoding Task.sleep in timeout/backoff makes those code paths untestable (tests block on real time). Inject any Clock<Duration> from day one; retrofitting a clock through the orchestrator and retry policy later is invasive.
- @Observable invalidates views on any tracked stored-property access. If injected services (pipeline, DB reader) are stored as plain properties they can trigger spurious view updates and retain cycles; mark them @ObservationIgnored.
- Keep the Connector protocol free of any concrete URLSession/SDK type in its signature. If a live network type leaks into fetch()'s signature, the future out-of-process/XPC runner becomes impossible without breaking the protocol — defeating the locked 'abstract enough to back with an out-of-process runner' requirement.

**Risks**
- [high] In Swift 6.2 / Xcode 26, new app targets default to SWIFT_DEFAULT_ACTOR_ISOLATION = MainActor, but SPM packages default to nonisolated and must opt in per-target via .defaultIsolation(MainActor.self). — SE-0466 (Control default actor isolation) plus Donny Wals and Use Your Loaf articles confirm new Xcode 26 projects apply implicit MainActor, while a new SPM package does NOT set the flag (defaults to nonisolated) and exposes .defaultIsolation as a per-target SwiftSetting requiring swift-tools-version 6.2.
- [high] GRDB 7.x exposes Sendable DatabaseReader/DatabaseWriter protocols (adopted by DatabaseQueue/DatabasePool) suitable as the persistence DI seam, with async methods taking @Sendable closures. — GRDB README/protocol references state DatabaseReader conforms to AnyObject and Sendable, DatabaseWriter is adopted by DatabaseQueue and DatabasePool, async methods use @Sendable Result<Database,Error> closures; latest release 7.10.0 (2026-02-15), Swift 6.1+.
- [high] A non-throwing outer withTaskGroup + per-child Result/outcome capture guarantees a slow/failing connector cannot cancel siblings, while a throwing outer group would cascade-cancel and rethrow. — SE-0304 and current TaskGroup guidance: if an error is thrown from the withThrowingTaskGroup closure, all unfinished child tasks are cancelled and the error rethrown; individual child throws do not affect the group unless propagated. Capturing per-child outcomes is the standard partial-results pattern.
- [medium] The same connector behavior survives the inner timeout race only if the connector's underlying network call honors cooperative cancellation; otherwise a hung socket keeps running after timeout. — Structured concurrency cancellation is cooperative; URLSession async APIs do cancel on Task cancellation, but custom transports must check Task.isCancelled. Inferred from concurrency semantics, not a single cited line for this exact app.
- [high] swift-dependencies and plain protocol/initializer injection are both production-viable in 2026; swift-dependencies is the de-facto community standard for ambient deps with scoped test overrides. — Point-Free swift-dependencies repo + multiple 2026 DI guides describe @Dependency/DependencyValues with withDependencies scoping; protocol-based injection remains the recommended baseline.
- [high] ASWebAuthenticationSession + PKCE is the correct native OAuth path for Google and Slack, and loopback-IP redirect is deprecated for some Google client types. — Google's 'OAuth 2.0 for iOS & Desktop Apps' notes loopback redirect deprecation for certain client types and PKCE support; AppAuth-iOS and Apple guidance use ASWebAuthenticationSession as the recommended flow; Slack UA-validation caveat surfaced in search results.
- [medium] The Connector protocol exchanging only Sendable Codable value types is sufficient to later back it with an out-of-process/XPC runner without changing call sites. — Architectural inference: XPC/subprocess proxies require serializable, Sendable payloads, which Codable value types satisfy. No external source specific to Daybrief verifies this; it is a sound but unverified design assertion.

**Adversarial verifications**
- **[CONFIRMED]** The same connector behavior survives the inner timeout race only if the connector's underlying network call honors cooperative cancellation; otherwise a hung socket keeps running after timeout.
  - Both technical pillars are confirmed by authoritative current sources for Swift 6.2 / macOS 26 (these semantics are stable since Swift 5.5 and unchanged in 6.2).

(1) COOPERATIVE CANCELLATION: Swift structured concurrency cancellation only sets the isCancelled flag; the runtime never forcibly stops a task. Code must voluntarily poll Task.isCancelled or call try Task.checkCancellation() at safe points and stop work itself. Sources: Apple WWDC23 "Beyond the basics of structured concurrency" (developer.apple.com/videos/play/wwdc2023/10170/), Swift with Majid "Task Cancellation in Swift Concurrency" (2025-02-11), Hacking with Swift "How to cancel a Task." Quote: "Swift will never stop your task automatically... It is totally up to you to decide how to handle this information."

(2) URLSESSION CANCELS IN-FLIGHT, CUSTOM TRANSPORTS DO NOT: URLSession's async APIs (data(for:delegate:), data(from:)) DO honor cooperative cancellation — cancelling the enclosing Swift Task cancels the underlying URLSessionTask while the request is still in flight. Apple DTS engineer Quinn "The Eskimo!" confirmed this on the Swift Forums with running test code (forums.swift.org/t/does-task-cancellation-propagate-to-urlsessiontasks/65041). A second forum thread (forums.swift.org/t/.../69218) shows a mid-flight request aborts immediately on Task.cancel(), throwing NSURLErrorDomain Code=-999 "cancelled" (URLError.cancelled). Conversely, any transport that does NOT check cancellation (raw/blocking socket, synchronous C library wrapper) gets no automatic interruption — it must poll Task.isCancelled or it keeps running.

The claim's conditional logic follows directly from (1)+(2): a connector on URLSession's async API survives the inner timeout race correctly (the in-flight call is aborted); a connector on a custom transport that ignores cancellation leaves the awaiter timing out while the orphaned socket keeps running — exactly the described failure. The evidence offered is honestly labeled as inferred from concurrency semantics; that inference is sound and matches documented framework behavior.
  - _caveat:_ Minor imprecision in the surrounding framing (not in the claim itself): a cancelled URLSession async call surfaces as URLError.cancelled (NSURLErrorDomain code -999), NOT CancellationError. This does not change whether the network call aborts. Also, "honors cooperative cancellation" must mean the transport actually reaches a suspension/checkpoint — a truly synchronous blocking syscall on a thread cannot be interrupted by cancellation no matter what, so for non-URLSession transports the developer must architect cancellation checkpoints (timeouts, polling) deliberately; cancellation alone never preempts a blocked thread.
- **[REFUTED]** The Connector protocol exchanging only Sendable Codable value types is sufficient to later back it with an out-of-process/XPC runner without changing call sites.
  - Sendable+Codable value payloads are NECESSARY for XPC/distributed transport, but NOT SUFFICIENT to preserve call sites. Call-site stability is determined by the protocol's INTERFACE SHAPE, which the claim never constrains. Evidence (current as of 2025-2026, macOS 14-26, Swift 6.x):

(1) XPC is inherently asynchronous and fallible. Every XPC mechanism forces async+throwing methods. NSXPCConnection requires all remote-interface methods to return void with reply blocks (no synchronous return values); the modern Swift XPCSession/XPCListener API (macOS 14+) sends one-shot Codable messages invoked via try/await. Apple DTS (Quinn) and the ChimeHQ/AsyncXPCConnection docs note all XPC calls can fail even when the method declares no error, and naive `async` XPC methods are unsafe re: connection-failure hangs. So an in-process Connector with synchronous, non-throwing methods (e.g. `func fetch() -> [Item]`) CANNOT be XPC-backed without rewriting signatures to `async throws` — which changes every call site.

(2) Codable/Sendable is the PAYLOAD requirement, not the INTERFACE requirement. Swift distributed actors (WWDC22 'Meet distributed actors', SE-0336/0344/0428) are the only Apple-designed 'location transparency' mechanism, and even there preserving call sites requires far more than Codable values: methods must be declared `distributed func`, the type must be a `distributed actor` (not a plain protocol/struct value-exchange), it must declare an associated ActorSystem with a SerializationRequirement, and remote references come from `resolve(id:using:)` not a normal initializer. Cross-actor distributed calls are ALSO implicitly async+throws, so the call site must already sit in an async/throwing context.

(3) Therefore the assertion is only true conditionally — if the Connector is ALREADY an async-throwing interface (or a distributed actor) AND its surface obeys the chosen transport's registration model. As an unconditional claim ('only Sendable Codable value types is sufficient ... without changing call sites') it is false: it conflates payload-type constraints (true) with interface-shape/async/throwing/registration constraints (the actual determinant, omitted).

Sources: Apple Developer Forums thread/749722 (XPCSession/XPCListener Codable, macOS 14, IncomingSessionRequest validation gap FB13715123); github.com/CharlesJS/SwiftyXPC and trilemma-dev/SecureXPC (Codable-based wrappers, async/throwing, error-domain registration); github.com/ChimeHQ/AsyncXPCConnection (XPC reply-callback not guaranteed; void-return requirement); developer.apple.com/videos/play/wwdc2022/110356 + swift-evolution SE-0336/0344/0428 (distributed func keyword, SerializationRequirement, implicit async/throws, resolve(id:using:)).
  - _caveat:_ The claim is DIRECTIONALLY sound and becomes TRUE if an unstated precondition holds: the in-process Connector protocol is already shaped as an async-throwing (or distributed-actor) interface whose methods would not need signature changes under an XPC/distributed transport. In that case, restricting exchanged data to Sendable+Codable value types is indeed the right enabling constraint and call sites can survive. I could not find any Daybrief-specific source; this verdict evaluates the general Swift/macOS technical assertion. If the team's actual Connector is defined with `async throws` methods from day one, downgrade this from 'refuted' to 'confirmed-with-conditions'. The weakness is specifically the words 'only ... value types is sufficient ... without changing call sites' as an unconditional guarantee.

## Licensing & contribution governance for an OSS-now / productize-later local-first macOS app (Daybrief)

**Summary.** Recommend AGPL-3.0-only as the outbound license PLUS a contributor CLA with broad license-back (Apache-style ICLA with an explicit relicensing grant), NOT a DCO. Rationale: AGPL is genuine OSI open source (best for a community-connector ecosystem and for the "private-by-default" brand), its §13 network clause does NOT burden desktop users of a local-first binary, yet it deters a competitor from launching a closed hosted fork — and a CLA preserves your unilateral right to dual-license a future commercial/hosted tier (which AGPL alone cannot give you). The screen-capture module is a product-policy/build-flag concern, not a license concern: AGPL does not force you to ship it anywhere, and AGPL's copyleft actually HELPS guarantee it never silently appears in a third party's closed hosted fork. BSL/FSL are viable fallbacks if you want time-boxed anti-free-riding without copyleft, but they are source-available (not open source), hurt connector-contributor goodwill, and you would STILL need a CLA to dual-license. SSPL is rejected: OSI-disqualified, reputationally toxic post-Redis-reversal, and pointless for a desktop app with no SaaS to protect today.


**Decisions**
- **Outbound license = AGPL-3.0-only (not 'or-later'), single-license public repo** — AGPL-3.0 is OSI-approved real open source, which matters for credibility with a privacy-focused, contributor-driven connector ecosystem and aligns with the local-first/private-by-default brand. Its §13 network clause requires only that a party who MODIFIES the code AND offers it over a network publish their modified source — verified to NOT trigger for ordinary desktop users running the local-first binary, so it imposes near-zero burden on end users while strongly deterring a competitor who wants to wrap the code in a closed hosted tier. 'AGPL-3.0-only' (pin the version) avoids an unbounded 'or-later' grant the FSF could change under you.
- **Governance = Contributor License Agreement (CLA) with a broad inbound license-back, NOT a DCO** — A DCO is only a certification of origin; it does NOT grant the project the right to relicense, so under a DCO every external contributor's copyright would block any future dual-license/commercial relicensing unless re-consented individually. A CLA that grants the project a perpetual, irrevocable license to use the contribution under AGPL AND any other license (incl. proprietary) — without requiring full copyright assignment — preserves the unilateral ability to stand up a commercial/hosted tier later. This is the load-bearing decision the spec §12 'decide before first external PR' is really about: relicensing rights cannot be retrofitted onto a DCO project.
- **Reject SSPL outright** — SSPL is OSI-rejected, was reputationally damaging enough that Redis adopted it in 2024 and reverted to AGPLv3 in 2025, and its entire purpose (forcing a cloud reseller to open-source their whole operational stack) is meaningless for a desktop app with no SaaS surface today. It would maximize contributor friction for zero benefit.
- **Reject BSL 1.1 and FSL as the primary choice; keep FSL as the only acceptable non-copyleft fallback** — Both are source-available (not OSI open source), which undercuts community-connector goodwill and excludes the code from 'open source' directories/distros. BSL 1.1 requires you to PARAMETERIZE (Additional Use Grant, Change Date ≤4yr, GPL-compatible Change License) — more drafting surface and ambiguity. FSL-1.1 is the cleaner of the two (fixed 2-year conversion to Apache-2.0 or MIT, fixed Competing-Use clause, patent grant) and is the right fallback IF the team prioritizes explicit anti-free-riding over OSI-open-source status. Both STILL require a CLA to dual-license, so they don't remove the governance decision.
- **Treat the opt-in screen-capture module as a build/product-policy boundary, not a licensing mechanism** — No license can say 'this module must never ship in a hosted tier'; that is enforced by NOT building it into the hosted artifact (compile-time flag / separate Swift target excluded from the server build) and by product policy. AGPL is positively helpful here: because any networked, modified fork must publish source, a competitor cannot quietly bolt screen-capture into a closed hosted clone without exposing it.

**Stack**
- `AGPL-3.0-only` (3.0 (2007)) — Primary outbound license. OSI-approved copyleft; §13 closes the hosted-fork loophole without burdening local-first desktop users; supports dual-licensing when paired with a CLA.
- `Apache-style ICLA with relicensing grant (e.g. CLA Assistant / cla-bot tooling)` (ICLA template + GitHub CLA bot) — Retains unilateral relicense/dual-license rights for a future commercial/hosted tier. Use a license-back grant (not full copyright assignment) to minimize contributor friction.
- `FSL-1.1-Apache-2.0 (fallback only)` (1.1) — If the team chooses time-boxed anti-free-riding over OSI status: fixed 2-year conversion to Apache-2.0, fixed Competing-Use clause, includes patent grant + termination. Still needs a CLA to dual-license.
- `GRDB.swift` (MIT) — Dependency license note: MIT is permissive and compatible with AGPL, BSL, FSL, or any proprietary outbound license — imposes no copyleft on Daybrief.
- `SQLCipher (Community Edition)` (BSD-style) — Dependency license note: BSD-style with an attribution/notice obligation (must reproduce the license grant in app materials). Compatible with any outbound choice; no copyleft. Commercial Edition only needed for Zetetic's enhanced features/support, not for licensing compliance.

**Patterns / concrete notes**
- AGPL §13 'network clause' is a two-pronged trigger: it requires source disclosure ONLY when a party (a) MODIFIES the program AND (b) offers that modified program to remote users over a network. A user running the unmodified local-first desktop binary triggers neither prong — so AGPL does not impose source-publishing or copyleft propagation on end users of Daybrief.
- AGPL §13 obligations attach to the PERSON OPERATING the network service, not to the upstream project. The upstream Daybrief repo just ships AGPL source; it has no ongoing AGPL burden of its own.
- FSL-1.1 'Competing Use' (verbatim from the license): making the Software available to others in a commercial product or service that (1) substitutes for the Software, (2) substitutes for any other product/service the licensor offers using the Software as of the availability date, or (3) offers the same or substantially similar functionality. Everything else is a 'Permitted Purpose' (internal use, non-commercial education, non-commercial research, professional services to a licensee).
- FSL-1.1 converts to Apache-2.0 (FSL-1.1-ALv2) or MIT (FSL-1.1-MIT) on the SECOND ANNIVERSARY of the date each version is made available — per-version, not per-project. It includes a patent grant with defensive termination (your patent license ends if you sue claiming infringement).
- BSL 1.1 is a fill-in-the-blanks template with three required parameters: Additional Use Grant (production carve-out, or 'None'), Change Date (must be ≤4 years from release), and Change License (must be GPLv2-or-later / GPL-compatible). On the earlier of the Change Date or the 4-year cap, the code auto-converts to the Change License.
- CLA mechanics for relicensing: a CLA that assigns the project a perpetual, worldwide, irrevocable license to relicense contributions under any terms is sufficient to enable dual-licensing — full copyright ASSIGNMENT is not required and is the higher-friction option. DCO ('Signed-off-by') grants no relicensing right at all.
- Dual-license operating model: publish the repo under AGPL-3.0; require CLA on all external PRs; for the future hosted/commercial tier, build from your own copy under a separate proprietary grant you hold via aggregated CLA rights. The screen-capture module is excluded from the hosted build target at compile time, never gated by license text.
- SQLCipher Community attribution: include the SQLCipher BSD notice in the app's about/acknowledgements screen and bundled licenses file regardless of outbound license — this is a standing obligation independent of the AGPL/FSL choice.

**Gotchas**
- A DCO cannot be 'upgraded' to relicensing rights later — if you accept the first external PR under a DCO and then decide to dual-license, you must individually re-secure permission from every external contributor or rip out their code. Decide CLA-vs-DCO BEFORE the first external merge (this is the real meaning of spec §12).
- AGPL alone does NOT let you sell a closed commercial version of contributors' code — you only own your own contributions. Without a CLA, AGPL + external contributions = you are permanently locked to AGPL for the combined work. The CLA is what makes the 'productize-later' path legal.
- No license clause can guarantee the screen-capture module 'never ships in a hosted tier' — that is a build-system and policy control (separate Swift target / compile flag), not something the LICENSE file can enforce.
- Choosing FSL/BSL means you CANNOT call Daybrief 'open source' truthfully (both are source-available / not OSI-approved). Mislabeling invites community backlash — material for a privacy-positioned project.
- SQLCipher Community Edition's BSD-style license carries an explicit attribution/reproduction-in-app obligation; forgetting the in-app notice is a compliance miss independent of the outbound license decision.
- 'AGPL-3.0-or-later' hands the FSF the ability to bind your project to a future AGPLv4 you haven't read; prefer 'AGPL-3.0-only' to keep the terms fixed.
- BSL's Change License must be GPL-compatible — you cannot make BSL convert to a permissive Apache/MIT; if you want permissive conversion you must use FSL, not BSL.
- AGPL can scare risk-averse corporate contributors/users (many companies blanket-ban AGPL). For a community-CONNECTOR ecosystem this is usually fine (contributors are individuals/enthusiasts), but it can deter enterprise adoption of a future SDK — weigh if connectors are meant to be embedded by third parties.

**Risks**
- [high] AGPL §13 does not impose copyleft/source-disclosure obligations on ordinary end users running the unmodified local-first desktop binary — Multiple authoritative analyses (Revenera, Mend, FOSSA/Heather Meeker, opensource.com) confirm §13 requires BOTH modification AND network offering of the modified program; a local desktop app served to no remote users triggers neither prong.
- [high] A CLA with a broad license-back (not full copyright assignment) is sufficient to retain unilateral dual-/re-licensing rights, whereas a DCO is not — opensource.com (CLA vs DCO), FINOS BOK, and Wikipedia CLA article all state DCO grants no relicensing right and that CLAs provide the clear chain enabling relicensing; copyright assignment is one (heavier) option, not a requirement.
- [high] FSL-1.1 converts to Apache-2.0/MIT on the second anniversary of each version and defines Competing Use exactly as quoted — Pulled the verbatim FSL-1.1-ALv2 template text from fsl.software and getsentry's repo via curl; 'Grant of Future License' specifies the second anniversary, and 'Permitted Purpose/Competing Use' clauses are quoted directly.
- [high] SSPL is OSI-rejected and was abandoned by Redis (SSPL 2024 -> AGPLv3 2025) — OSI publicly rejected SSPL; multiple 2025-2026 sources (DEV, opensourcealternatives, license guides) report Redis's SSPL adoption in 2024 and reversion to AGPLv3 in 2025.
- [high] GRDB.swift is MIT and SQLCipher Community is BSD-style, both compatible with any outbound license choice for Daybrief — GRDB.swift LICENSE on GitHub (groue/GRDB.swift) = MIT; Zetetic's SQLCipher license page states Community Edition is BSD-style with an attribution requirement.
- [medium] BSL 1.1's Change License must be GPL-compatible (cannot convert to permissive Apache/MIT) — FOSSA and MariaDB BSL FAQ state the Change License is mandated to be GPL v2-or-later or GPL-compatible; some readings allow GPL-compatible permissive licenses, so 'cannot be Apache/MIT' is the conservative interpretation — verify against the exact MariaDB template wording before relying on it.
- [medium] AGPL meaningfully DETERS (does not legally prohibit) a competitor from launching a closed hosted fork — AGPL forces any networked modified fork to publish source, which removes the competitive moat of a closed clone; however a determined competitor could run an unmodified or source-published fork, so deterrence is economic/practical, not absolute — FSL/BSL's Competing-Use bar is a stronger legal prohibition but at the cost of OSI-open-source status.

**Adversarial verifications**
- **[REFUTED]** BSL 1.1's Change License must be GPL-compatible (cannot convert to permissive Apache/MIT)
  - The first half is TRUE but the parenthetical is FALSE. The BSL 1.1 template (authored by MariaDB) requires the Change License to be "the GPL Version 2.0 or any later version, or a license that is compatible with GPL Version 2.0 or a later version, where 'compatible' means that software provided under the Change License can be included in a program with software provided under GPL Version 2.0 or a later version" (verbatim from https://mariadb.com/bsl11/, corroborated by https://fossa.com/blog/business-source-license-requirements-provisions-history/ and https://mariadb.com/bsl-faq-mariadb/). So "must be GPL-compatible" is correct.

But "cannot convert to permissive Apache/MIT" is wrong. The FSF (https://www.gnu.org/licenses/license-list.en.html) classifies the Expat/MIT License as "a lax, permissive non-copyleft free software license, compatible with the GNU GPL." MIT is permissive AND GPL-compatible, so it is a perfectly valid BSL 1.1 Change License — directly contradicting the claim that permissive licenses are excluded. (Indeed, real-world BSL adopters such as Sentry have used permissive Change Licenses.) The claim conflates "permissive" with "GPL-incompatible," which is false.

Apache 2.0 is the only nuance: FSF and Apache (https://www.apache.org/licenses/GPL-compatibility.html) agree Apache 2.0 is NOT compatible with GPLv2, but it IS compatible with GPLv3. Because the BSL template explicitly permits "GPL Version 2.0 OR a later version," Apache 2.0 (being GPLv3-compatible) also plausibly satisfies the requirement, so even Apache is not clearly excluded.
  - _caveat:_ The core requirement (Change License must be GPL-compatible) is solidly confirmed against the exact MariaDB template text. The refutation targets the parenthetical only. License compatibility is ultimately a legal determination — the FSF classification of MIT as GPL-compatible is authoritative for the FSF's interpretation, and the BSL text's own definition of "compatible" (can be combined with GPLv2+ in one program) is also satisfied by MIT. This is general licensing analysis, not legal advice; for a high-stakes choice consult counsel and the specific licensor's BSL header, since licensors may also offer additional non-GPL-compatible Change License alternatives beyond the mandatory GPL-compatible baseline.
- **[CONFIRMED]** AGPL meaningfully DETERS (does not legally prohibit) a competitor from launching a closed hosted fork; the FSL/BSL Competing-Use bar is a stronger legal prohibition but costs OSI-open-source status.
  - Every leg checks out against current authoritative sources (June 2026).

(1) AGPL deters, does not prohibit: AGPLv3 §13 closes the SaaS/ASP "loophole" left in GPLv3 — a networked MODIFIED version must offer its Corresponding Source to remote users (FSF AGPLv3 text; FOSSA/Mend/Revenera glossaries). This strips away the secret-modifications moat of a closed clone. But AGPL is OSI-approved free software and explicitly permits commercial use and competing services. danb.me "Common AGPL Misconceptions" is decisive: AGPL CANNOT carry a no-compete/SaaS restriction because §7 does not allow that category of additional restriction ("do not fit into the allowed categories... so can again be removed"), and source need only go "to those you distribute to," not the public at large. So a competitor may legally launch a hosted fork — they must merely publish modified source to their service's users. Deterrence is economic/practical, not legal/absolute. The DEV "Open Source in 2026: Fork Wars" piece and Plausible/Lago AGPL-adoption posts corroborate AGPL being chosen specifically as an anti-free-riding deterrent, not a prohibition.

(2) Escape hatch confirmed: running unmodified (no new disclosure beyond upstream) or running a source-published modified fork are both lawful, so a determined competitor is not barred — matching "deterrence is not absolute."

(3) FSL/BSL = stronger LEGAL bar: FSL-1.1 (Sentry, Nov 2023) template grants rights only for a "Permitted Purpose" that EXPRESSLY EXCLUDES "Competing Use" — defined as making the software available in a commercial product/service that substitutes for it or "offers the same or substantially similar functionality." That is an affirmative prohibition for the term (FSL: 2 yrs to Apache-2.0/MIT; BSL: up to 4 yrs), not economic friction. Confirmed via the FSL-1.1-MIT license template and Sentry/Heather Meeker analyses.

(4) Cost = OSI status: FSL/BSL are "source-available"/"fair source," NOT OSI-approved open source, because the field-of-use/competing-use restriction violates the Open Source Definition (no discrimination against fields of endeavor). Confirmed by fsl.software, Heather Meeker, The Register, and acrosslegal.

Sources: FOSSA (fossa.com/blog/open-source-software-licenses-101-agpl-license), Mend (mend.io/blog/the-saas-loophole-in-gpl-open-source-licenses), danb.me/blog/common-agpl-misconceptions, fsl.software + getsentry FSL-1.1-MIT template, blog.sentry.io/introducing-the-functional-source-license, heathermeeker.com (FSL launch), theregister.com 2023/11/24 opinion.
  - _caveat:_ Two precision points. (a) AGPL's deterrent strength is partly overstated in common framing: the source-disclosure duty runs only to the service's ACTUAL users, not the public, so a competitor with few/controlled users faces weaker practical friction than "must open-source your whole backend publicly" implies; conversely, enterprises' blanket AGPL bans ADD non-legal deterrence beyond the license text itself. (b) FSL/BSL prohibitions are time-limited (2 yrs FSL / up to 4 yrs BSL) and then auto-convert to permissive open source, so the "stronger prohibition" is temporary, not perpetual — and is contractual/copyright-grant-scoped, whose enforceability against a non-licensee is less battle-tested in court than AGPL's copyleft.
